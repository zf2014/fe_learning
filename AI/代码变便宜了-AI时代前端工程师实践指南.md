## 原文翻译

### 代码变便宜了（更便宜了）

**作者：** Carson Gross
**日期：** 2026年6月4日

一个不可否认的事实是，在过去的一年里，编写代码的成本大幅降低。人工智能能够以惊人的速度生成大量、往往质量尚可的代码。装作这种情况不存在是没有意义的。

有时，面对这个确实令人不舒服的事实，我看到我尊敬的人会说类似这样的话："写代码从来都不是问题。"

虽然我理解这种情绪，但我并不完全同意：当然，写代码至少是问题的一部分。随着有效AI编码工具的出现，这一部分问题已经显著缩小。

那么，原始编码的重要性降低，对软件开发者——那些过去以（并经常比较）自己编码能力为傲的人——意味着什么呢？

### 理解更贵了（更贵了）

我看到的一个影响是，理解代码变得更加昂贵。这是因为当大量代码被生成，而不是从某个程序员的指尖痛苦地涌现时，没有人理解那些代码。

如果说对这些代码的理解必须存在，那么这种理解必须是在代码编写之后，通过阅读代码来完成。请注意，传统的观点是，阅读别人的代码比写自己的代码更难。

一些AI爱好者说："谁在乎呢，你也不理解编译器的输出。"

我认为这在多个方面都是范畴错误：

- 编译器是确定性的；而LLM（大语言模型）从设计上就不是确定性的。
- 编译器工作流程保留原始源代码；而LLM工作流程通常不会。
- 编译器的输出范围狭窄（机器代码）；而LLM的输出则不是（通用软件）。

我坚持认为，在大多数情况下，对于关键任务软件，开发者仍然需要理解底层代码，即使它是由LLM生成的。

如果代码是由LLM生成的，还有一个明显的危险：LLM生成代码的速度可能远远超过你或其他人理解它的速度。因此，我建议增量使用LLM，而不是允许它们生成庞大的、你或任何人都无法理解的变更列表。

（在有些情况下这样做是合适的，例如机械式重构，但在向代码库引入新的语义时，这极其危险。）

### 魔法师学徒的陷阱

在观看AI获得越来越多关注的过程中，有一个电影场景一直在我脑海中浮现，那就是迪士尼电影《幻想曲》中的《魔法师学徒》。

在这个场景中，学徒决定用魔法来协助完成打扫的苦差事。他对一把扫帚施加了魔法，扫帚随后开始打扫一切。起初一切看起来都很顺利，直到扫帚越来越用力地打扫，达到了字面意义上的"让一切漂起来"的程度。

当魔法师重新出现并对局面施加控制，愤怒地盯着学徒的愚蠢时，混乱才得以解决。

这似乎是AI时代的一个恰当隐喻：你想成为一名魔法师，而不是学徒。

而魔法师必须理解代码。

### 复杂性：依然不好

人类通常对几何曲线和指数曲线的把握很差。
（这就是他们相信复利这样的童话的原因。）

代码变便宜的核心危险是复杂性。我在没有证据的情况下断言，复杂性倾向于至少以几何方式，经常以指数方式随着系统规模增长。

在LLM之前，就有多产的人类程序员。
也许你和这样的人共事过：他们能写很多代码。

我见过那些缺乏对复杂性应有的恐惧的多产程序员，他们在现有问题上堆砌越来越多的代码，直到整个系统坍塌成无法修改的稳定状态，任何更改都会引入与修复一样多的错误。

LLM无法恐惧复杂性，而且它们是多产的程序员。
对我来说这似乎很危险。

### 减法、约束型工程师

为了解决LLM生成代码的危险，我提出了减法、约束型工程师：

这种工程师说"不"，仔细审查LLM的输出，建议简化方案，并在处理LLM生成的代码时总体保持强有力的控制。

他们不以自己创建的代码为傲，而以从系统中移除或阻止进入系统的代码（和层）为傲。

这种精神更像是雕塑家，而不是建筑师。

建筑师的精神在某种程度上仍然适用的地方是在系统设计层面：一个好的工程师需要知道如何有效地组合组件来创建系统。然而，即使在这里，我认为减法思维也会很有用：移除不必要的组件和系统边界，以简化系统部署和组件间交互等。

减法型工程师与大多数过去是程序员的人是不同类型的工程师。我承认我一直对减法型工程师的思维模式感到同情：我不介意说不，我不介意改进现有系统而不是进行英雄式的重写等。

但承认我自己的偏见，我认为这是与LLM互动的一种富有成效的方式，它保留了计算机编程的艺术，并正确承认了一个双重现实：代码的创建成本已经大幅降低，而复杂性仍然是我们的顶级威胁。

## 核心观点提炼

### 三个核心洞察

1. **代码生成变便宜了**
   - AI能够快速生成大量代码
   - 代码质量往往尚可
   - 这是一个无法否认的事实

2. **理解代码变贵了**
   - AI生成的代码缺乏原始理解
   - 理解必须在代码生成后通过阅读完成
   - 阅读别人的代码比写自己的代码更难

3. **复杂性是核心威胁**
   - 复杂性倾向于几何/指数级增长
   - LLM无法恐惧复杂性
   - LLM是多产的程序员，这很危险

### 关键隐喻：魔法师 vs 学徒

- **学徒**：使用魔法（AI）但不理解，导致失控
- **魔法师**：理解魔法（代码）并能控制它

**结论：你想成为一名魔法师，而不是学徒。**

## 传统前端开发 vs AI时代开发对比

### 传统开发方式的问题

| 维度 | 传统方式 | 问题 |
|------|---------|------|
| **技能重心** | 写代码速度、框架熟练度 | 在AI时代这些价值降低 |
| **复杂管理** | 缺乏系统性考虑，随需求堆叠 | 复杂性呈几何级增长 |
| **技术选型** | 倾向于引入新库/新框架 | 增加系统边界和复杂性 |
| **代码审查** | 松散或缺失 | 面对AI生成的大量代码更危险 |
| **架构思维** | 加法思维（增加功能/组件） | 导致系统臃肿、难以维护 |

### AI时代开发方式的优势

| 维度 | AI时代方式 | 优势 |
|------|-----------|------|
| **技能重心** | 代码理解、架构设计、约束管理 | 在AI时代价值提升 |
| **复杂管理** | 系统性考虑，主动控制复杂性 | 保持系统可控 |
| **技术选型** | 最小化依赖，质疑必要性 | 降低系统边界和复杂性 |
| **代码审查** | 严格审查，深度理解 | 确保AI生成代码可控 |
| **架构思维** | 减法思维（删除冗余、简化系统） | 保持系统简洁可维护 |

## 前端工程师需要做出的6大调整

### 1. 从"快速实现"转向"渐进式审查"

#### 传统做法
```
需求 → 立即编码 → 交付
```

#### AI时代建议
```
需求 → 小范围AI生成 → 深度审查 → 渐进扩展 → 持续理解
```

#### 具体实践
- 不要让AI一次性生成整个模块，而是分小块生成
- 每次生成后必须先理解代码逻辑，再继续
- 建立"AI输出审查清单"（安全性、性能、可维护性）

#### 审查清单示例
```markdown
## AI生成代码审查清单

### 逻辑正确性
- [ ] 是否实现了预期的功能？
- [ ] 是否考虑了所有边界情况？
- [ ] 错误处理是否完善？

### 代码质量
- [ ] 是否符合项目代码规范？
- [ ] 是否存在代码重复？
- [ ] 变量和函数命名是否清晰？

### 性能考虑
- [ ] 是否存在性能瓶颈？
- [ ] 是否有不必要的重渲染？
- [ ] 内存使用是否合理？

### 安全性
- [ ] 是否存在XSS/CSRF风险？
- [ ] 敏感数据是否正确处理？
- [ ] 输入验证是否充分？

### 可维护性
- [ ] 代码逻辑是否容易理解？
- [ ] 是否需要添加注释？
- [ ] 是否过度抽象？
```

### 2. 从"技术栈广度"转向"架构深度"

#### 传统做法
- 追求学习更多框架（React、Vue、Svelte、Solid...）
- 热衷引入新库（UI库、状态管理、路由...）

#### AI时代建议
- **深入理解核心原理**：而不是框架API
- **掌握组件化设计模式**：高内聚、低耦合
- **理解前端架构模式**：模块化、微前端、Monorepo

#### 关键能力对比

```typescript
// 传统：会用就行
import { useState } from 'react'
const [count, setCount] = useState(0)

// AI时代：理解原理
// - 为什么需要useState？（闭包陷阱）
// - React的渲染机制如何工作？
// - 什么时候应该使用useReducer？
// - 如何避免不必要的重渲染？
// - 如何正确使用useEffect的依赖数组？
// - 如何优化组件性能？
```

#### 深度理解示例

```typescript
// 不仅会用，还要理解背后的原理
const UserProfile = ({ userId }) => {
  // ❌ 传统：只知道怎么用
  const [user, setUser] = useState(null)

  // ✅ AI时代：理解为什么这样用
  const user = useUser(userId) // 自定义hook，封装逻辑

  // 理解点：
  // 1. 为什么需要自定义hook？（复用逻辑、分离关注点）
  // 2. 什么时候应该使用useMemo/useCallback？
  // 3. 如何避免不必要的重渲染？
  // 4. 状态应该提升到什么层级？
  // 5. 如何处理异步状态？
  // 6. 如何设计组件接口？

  return <div>{user?.name}</div>
}
```

### 3. 从"加法思维"转向"减法思维"

#### 文章核心概念："减法、约束型工程师"

##### 传统前端问题
- 新功能 → 新组件 → 新状态 → 新逻辑
- 库越来越多（bundel size膨胀）
- 组件层级越来越深

##### 减法思维实践
- **定期审查和删除不用的代码**（不是注释，是真删）
- **合并相似组件**（避免过度抽象）
- **减少状态管理复杂度**（能用本地状态就不用全局状态）
- **质疑每个依赖的必要性**（"这个库真的需要吗？"）

##### 代码示例对比

```typescript
// ❌ 加法思维：为每个小功能引入新库
import { useAuth } from 'auth-lib'
import { useTheme } from 'theme-lib'
import { useAnalytics } from 'analytics-lib'
import { useNotifications } from 'notification-lib'

const Dashboard = () => {
  const auth = useAuth()
  const theme = useTheme()
  const analytics = useAnalytics()
  const notifications = useNotifications()

  // 4个独立的库，4个依赖，4个学习成本
}

// ✅ 减法思维：合并和简化
const useAppContext = () => {
  // 一个hook管理所有共享状态
  return {
    auth: useAuth(),
    theme: useTheme(),
    analytics: useAnalytics(),
    notifications: useNotifications()
  }
}

const Dashboard = () => {
  const context = useAppContext()

  // 单一入口，清晰的依赖关系
}
```

##### 减法思维实践清单

```markdown
## 减法思维实践清单

### 日常审查
- [ ] 每周检查未使用的导入和变量
- [ ] 每周删除至少50行无用代码
- [ ] 每月审查并删除过时的功能

### 组件设计
- [ ] 组件是否过度拆分？
- [ ] 能否合并相似的组件？
- [ ] Props是否过多（超过7个）？

### 状态管理
- [ ] 能否用本地状态替代全局状态？
- [ ] 状态是否过度嵌套？
- [ ] 能否用computed值替代派生状态？

### 依赖管理
- [ ] 每个依赖的必要性是否已验证？
- [ ] 能否用原生API替代某个库？
- [ ] Bundle size是否在增长？
```

### 4. 从"依赖AI"转向"AI作为放大器"

#### 危险倾向（魔法师学徒陷阱）

```
"让AI生成整个登录模块，我就能快速交付了"
→ 生成1000行代码
→ 不理解内部逻辑
→ 出问题时束手无策
```

#### 正确做法
- **AI生成 = 第一版草稿**
- **你负责 = 理解、优化、约束**
- **AI擅长 = 模板代码、重复性工作**
- **你擅长 = 架构设计、复杂逻辑、边缘情况**

#### 工作流示例

```
1. 你设计好接口和数据流
   ↓
2. 让AI生成基础组件
   ↓
3. 你审查并理解每一行
   ↓
4. 你优化性能和可维护性
   ↓
5. 你添加边缘情况处理
   ↓
6. 你编写测试确保理解正确
   ↓
7. 你添加文档解释设计意图
   ↓
8. 提交（包含审查记录）
```

#### AI使用最佳实践

```typescript
// ❌ 错误：让AI生成整个功能
// "帮我写一个完整的用户登录功能，包括表单验证、API调用、错误处理"

// ✅ 正确：分步骤使用AI

// 步骤1：设计接口
interface LoginForm {
  email: string
  password: string
}

interface LoginResponse {
  token: string
  user: User
}

// 步骤2：让AI生成基础表单结构
// "基于LoginForm接口，生成一个React表单组件的基本结构"

// 步骤3：审查并理解AI生成的代码
// - 是否符合我的设计意图？
// - 是否有不符合项目规范的地方？
// - 是否可以简化？

// 步骤4：优化和添加细节
// - 添加表单验证逻辑
// - 添加错误处理
// - 添加加载状态
// - 优化性能（防抖、节流等）

// 步骤5：编写测试
// - 确保理解正确
// - 覆盖所有边界情况
```

### 5. 从"功能导向"转向"复杂性管理"

#### 传统前端KPI
- 功能交付数量
- 开发速度
- 代码行数（❌ 错误指标）

#### AI时代应该关注
- **代码可理解性**（自己和他人能快速看懂）
- **系统简单性**（能用10行解决的绝不用100行）
- **删除率**（删除了多少冗余代码）
- **测试覆盖率**（证明你理解了代码行为）

#### 复杂性管理工具

```typescript
// 定期做"健康检查"
- 组件层级深度（超过5层考虑重构）
- 循环依赖检测
- 未使用代码扫描
- Bundle size分析
- 复杂度度量（圈复杂度、认知复杂度）
- 技术债务跟踪
```

```markdown
- [ ] 测试覆盖率是否达标？
- [ ] 测试是否易于理解？
- [ ] 测试是否过于脆弱？
```

### 6. 从"个人英雄"转向"系统思维"

#### 传统前端开发
- "我来快速实现这个功能"
- 个人技术栈偏好主导决策
- 缺乏系统性考虑

#### AI时代需要
- **系统设计能力**（组件如何组合成系统）
- **约束意识**（知道什么时候说不）
- **长期维护视角**（不仅仅是快速交付）

#### 具体实践

```typescript
// 在接受AI生成代码前问自己：
1. 这段代码是否符合现有架构模式？
2. 会引入多少新依赖？
3. 增加了多少系统边界？
4. 团队其他成员能快速理解吗？
5. 6个月后还能维护吗？

如果任何一个答案是否定的 → 拒绝或重构
```

#### 系统思维决策框架

```markdown
## 系统思维决策框架

### 技术选型决策
- 问题是什么？
- 最小解决方案是什么？（不用新库能解决吗？）
- 新引入的复杂性值得吗？
- 长期维护成本如何？
- 团队学习成本如何？
- 有更简单的替代方案吗？

### 架构设计决策
- 如何组织组件以最小化依赖？
- 如何设计状态管理以避免复杂性？
- 如何划分系统边界以保持清晰？
- 如何设计接口以提高可测试性？

### 代码审查决策
- 代码是否增加了不必要的复杂性？
- 是否有更简单的实现方式？
- 是否符合系统整体架构？
- 是否考虑了长期维护性？
```

#### AI时代流程

### 项目结构改进

#### 传统结构
```
src/
├── components/  (所有组件混在一起)
├── utils/       (所有工具函数)
├── hooks/       (所有hooks)
└── pages/       (所有页面)
```

#### AI时代建议（减法思维）
```
src/
├── core/           (核心、不变的)
│   ├── components/ (基础UI组件)
│   ├── hooks/      (通用hooks)
│   └── utils/      (纯函数工具)
├── features/       (按功能模块划分)
│   ├── auth/       (认证相关，自包含)
│   │   ├── components/
│   │   ├── hooks/
│   │   ├── utils/
│   │   └── types/
│   └── user/       (用户相关，自包含)
│       ├── components/
│       ├── hooks/
│       ├── utils/
│       └── types/
├── shared/         (共享资源，最小化)
│   ├── types/      (类型定义)
│   ├── constants/  (常量)
│   └── api/        (API客户端)
└── App.tsx         (应用入口)
```

#### 为什么这样更好？

1. **按功能组织，减少跨模块依赖**
   - 每个feature自包含
   - 明确的模块边界
   - 易于理解功能范围

2. **每个feature自包含，便于理解和删除**
   - 删除功能时只需删除对应目录
   - 功能迁移时整个目录移动
   - 独立测试和部署

3. **core层稳定，features层可迭代**
   - 核心功能不易变化
   - 业务功能频繁迭代
   - 清晰的分层架构

## 量化指标：如何衡量改进效果

### 传统目标 vs AI时代目标

| 指标 | 传统目标 | AI时代目标 | 衡量方式 |
|------|---------|-----------|---------|
| **代码行数** | 越多越好 | 越少越好 | 代码行数统计 |
| **组件数量** | 细粒度拆分 | 合理拆分，避免过度 | 组件数量统计 |
| **依赖数量** | 使用最新库 | 最小化依赖 | package.json分析 |
| **审查时间** | 尽量快 | 充分理解后再通过 | Code review时间 |
| **删除率** | 几乎不删 | 定期清理 | 删除代码行数/新增代码行数 |
| **测试覆盖率** | 基本覆盖 | 高覆盖率，证明理解 | 测试覆盖率报告 |
| **文档质量** | 可选 | 必需，解释设计意图 | 文档完整性检查 |

## 总结：AI时代的前端工程师画像

### 传统前端工程师
```
快速编码者
框架熟练工
功能实现机器
```

### AI时代前端工程师
```
代码理解者
架构设计师
复杂性管理者
减法工程师
系统约束者
```

### 核心转变

| 转变维度 | 传统思维 | AI时代思维 |
|---------|---------|-----------|
| **价值来源** | 写代码的速度和数量 | 理解和管理代码的能力 |
| **工作重点** | 快速实现功能 | 保持系统简单可维护 |
| **技术策略** | 引入最新技术栈 | 最小化依赖，简化系统 |
| **质量标准** | 功能正常工作 | 代码可理解、可维护 |
| **角色定位** | 个人英雄 | 系统思考者 |

### 记住文章最后的话：

> **"代码已经变得更便宜，而复杂性仍然是我们顶级威胁。"**

前端工程师的新价值不在于写代码，而在于**管理代码生成的后果**，确保系统在AI时代依然保持简单、可理解和可维护。
