# Coding is solved? Software is not. — 文章深度分析

> 原文：[Arcplane Journal - Coding is solved? Software is not.](https://arcplane.ai/journal/software-is-not-solved)
> 作者：G Gao (Arcplane 团队) · 2026年5月19日
> 来源：TLDR Newsletter 推荐

---

## 一、文章核心观点

这篇由 Arcplane 团队发布的文章，围绕 Claude Code 创始人 Boris Cherny "编程已被基本解决"的论断展开反驳与延伸。

### 核心论点

**编码（coding）≠ 软件开发（software development）。AI 让写代码变快了，但写代码从来不是软件开发中最难的部分。**

文章用 **"熵减"（entropy reduction）** 这个比喻贯穿全文：

1. **软件开发是将混乱的意图逐步收敛为可验证变更的过程**。从模糊的产品需求 → 产品思考缩窄范围 → 设计赋予形态 → 实现转化为代码 → 审查与部署闭环，每一步都在降低不确定性。

2. **AI 编码反而可能增加熵**。AI 生成的测试套件可能庞大但只是验证了自己选择的实现；审查意见可能很长却在核心问题边缘打转；方案看起来很深思熟虑但关键产品决策仍未做出。这就是 "AI slop"——看起来完整但并未真正减少混乱。

3. **瓶颈已经转移**。代码来得更快了，但团队不一定更快地信任结果。问题从"能不能实现"变成了"实现得对不对、该不该做"。

### 四个反复出现的痛点

- **上下文（Context）**：不是越多越好，而是要刻意选择关键的上下文
- **规格说明（Specs）**：模糊需求对 AI 比对人类更危险——人会犹豫，AI 会自信地执行错误理解
- **可验证的证据（Evidence）**：代码生成成本下降时，信任成为最昂贵的部分
- **人为检查点（Human Checkpoints）**：某些时刻仍需要人的判断力，尤其是实现前的范围决策

### 文章的最终落脚点

文章最终引出了 Arcplane 的产品定位——为 AI agent 生成的代码提供真正的生命周期管理，而非"聊天记录到 diff"的无结构交接。

## 二、从前端角度的评价

从前端工程师的视角来看，文章的观点**非常精准但描述偏后端**。前端的现实更加复杂：

### 1. "编码已解决"在前端更难成立

前端编码的挑战不仅仅是"写逻辑"，还包括：
- **视觉还原的模糊性**：设计稿到代码没有唯一正确答案，间距、动效、响应式断点存在大量"品味"决策
- **跨浏览器/设备的隐性复杂度**：一个看似简单的 CSS 变更可能在 Safari 上表现完全不同
- **交互状态爆炸**：loading、error、empty、edge case 的 UI 状态远多于后端 API 的 happy path

AI 可以生成一个组件，但"这个 hover 动效感觉对不对"、"这个断点下的布局是否合理"——这些判断 AI 目前无法替代。

### 2. "熵减"比喻在前端尤为贴切

前端的混乱来源更多：
- 设计稿本身可能自相矛盾（移动端和桌面端逻辑冲突）
- 产品需求可能隐含大量未说明的交互状态
- 无障碍（a11y）、性能、SEO 等非功能性约束容易被忽略

AI 生成的前端代码往往**看起来**很完整（视觉上能跑），但状态管理混乱、可访问性缺失、边界情况未处理——这正是文章说的"看起来完成了但并未减少混乱"。

### 3. "Spec 的重要性"对前端影响最大

后端可以靠接口契约约束，但前端的需求往往以 Figma 截图 + 口头描述的形式存在。文章强调的 "spec should stay with the work" 在前端意味着：
- 组件的交互规格需要明确文档化（而非藏在设计师脑子里）
- 响应式行为、状态转换、无障碍要求需要提前定义
- 否则 AI 会自信地生成一个"看起来对但行为不对"的 UI

### 4. 前端对"人为检查点"的需求更强烈

前端代码的审查不仅仅是逻辑正确性，还包括：
- **体验审查**：动效是否自然、交互是否直觉
- **设计一致性**：是否与现有设计系统一致
- **性能影响**：是否引入了不必要的重渲染、包体积膨胀

这些判断高度依赖人的审美和经验，目前难以自动化。

## 三、职业规划上的启发

### 1. 从"代码实现者"转向"意图定义者"

如果编码确实在被逐步商品化，价值会向产业链上游转移——谁能更准确地**定义要做什么**，谁就更有价值。这意味着：
- 产品思维比纯技术深度更重要
- 能把模糊需求转化为清晰规格的能力是核心竞争力
- "理解问题"比"写解决方案"更稀缺

### 2. 审查和判断力是抗 AI 替代的壁垒

文章反复强调的"信任"问题暗示：未来最重要的能力之一是**判断代码/方案是否正确的能力**，而非生成代码的能力。培养方向：
- 系统设计能力（判断方案的架构合理性）
- 产品判断力（判断功能是否符合用户需求）
- 安全与质量意识（判断变更的风险边界）

### 3. "AI 工作流设计"是新赛道

Arcplane 本身就是一个信号：围绕 AI 编码的工作流工具正在成为一个品类。相关的职业机会包括：
- AI agent 工作流设计
- 自动化测试与验证体系设计
- 代码审查流程优化
- 技术规范与上下文管理

### 4. 全栈视野的价值在上升

当实现成本下降时，能够**横跨产品、设计、工程**的人变得更高效——他们可以独立完成从需求到交付的完整循环，用 AI 加速实现环节。纯手写代码的"深度专家"如果不提升视野广度，可能面临议价能力下降。

### 5. 领域专精仍然安全（但需要重新定位）

文章作者的身份（auth 产品团队）暗示：在安全、权限、合规等高风险领域，人类判断力仍然不可替代。关键是定位自己为**"在特定领域拥有判断力的决策者"**，而非"在特定领域写代码的人"。

---

## 四、总结

> **代码生产正在被商品化，但软件工程中"定义正确性"和"验证正确性"的工作不仅没有减少，反而因为 AI 的"自信的错误"变得更重要了。**

职业规划应向这两个方向倾斜：**定义正确的事** 和 **验证事情做对了**。
