# 技能工作流参考手册

> 基于你已安装的 **41 个技能**，按不同开发场景设计的最佳工作流
> 工作类型：全栈前端（新项目、加功能、重构、修 Bug、构建工具、API 对接）
> 任务管理：本地文件 / TODO
> DDD 了解程度：有一定概念但未实践

---

## 🌐 全景总览

```
                    ┌──────────────────────────────────────────┐
                    │          入口判断（该用哪个工作流？）       │
                    │                                          │
                    │  ● 全新功能/项目 → 工作流A （完整设计）    │
                    │  ● 修 Bug         → 工作流B （快速修复）  │
                    │  ● 重构代码       → 工作流C （安全改造）   │
                    │  ● 性能优化       → 工作流D （专项审计）   │
                    │  ● 理解代码库     → 工作流E （探索学习）   │
                    │  ● 对接 API       → 工作流F （代码生成）   │
                    │  ● 配置问题       → 工作流G （工具调试）   │
                    │  ● 小改动         → 工作流H （轻量速攻）   │
                    └──────────────────────────────────────────┘
```

## 工作流 A — 全新功能/项目（完整设计）

**适用场景**：从零开始的 Feature、新模块、新项目

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  Step 1: 领域设计访谈                                             │
│  ─────────────────                                             │
│  👉 /grill-with-docs                                             │
│                                                                  │
│  做什么：                                                         │
│  - AI 会反复追问你的设计决策                                       │
│  - 精炼术语，建立共享语言（Ubiquitous Language）                   │
│  - 自动生成/更新 CONTEXT.md（术语表）                             │
│  - 对非显而易见的决策生成 ADR                                      │
│                                                                  │
│  什么时候切到下一步：                                              │
│  遇到"光说看不清楚"的问题 → 切 Step 2                             │
│  设计足够清晰了         → 跳到 Step 4                             │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 2: 高保真问题 → 原型                                        │
│  ───────────────────────────                                     │
│  👉 /handoff → /prototype → /handoff 回到 Step 1                  │
│                                                                  │
│  做什么：                                                         │
│  - /handoff 把当前上下文打包                                      │
│  - /prototype 构建一次性原型                                      │
│    ├─ Logic 分支：终端程序测试状态机/业务逻辑                      │
│    └─ UI 分支：多个视觉变体切换看效果                              │
│  - 原型回答完问题后，/handoff 回 grill 会话继续                     │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 3: 生成 PRD                                               │
│  ──────────────                                                  │
│  👉 /to-prd                                                      │
│                                                                  │
│  做什么：                                                         │
│  - 自动综合当前对话中的所有决策                                    │
│  - 输出到本地文件（因为你不用 Issue Tracker）                      │
│  - PRD 中包含：问题陈述、模块划分、术语表引用、ADR 引用             │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 4: 制定实现计划                                             │
│  ───────────────────                                              │
│  👉 writing-plans                                                 │
│                                                                  │
│  做什么：                                                         │
│  - 拆解为具体的 TODO 任务                                         │
│  - 定义每个任务的验收标准                                          │
│  - 标注任务依赖关系                                                │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 5: 执行实现                                                 │
│  ────────────                                                    │
│  👉 subagent-driven-development 或 /start-work                     │
│                                                                  │
│  执行时每个任务遵循：                                              │
│  test-driven-development → 实现 → verification-before-completion   │
│                                                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Step 6: 完成收尾                                                 │
│  ────────────                                                    │
│  👉 finishing-a-development-branch                                │
│     requesting-code-review                                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

**完整命令链**（一次执行）：
```
/grill-with-docs
  → 遇到高保真问题: /handoff → /prototype 建原型 → /handoff 回来
  → 设计清晰: /to-prd
  → writing-plans
  → subagent-driven-development 或 /start-work
  → finishing-a-development-branch
```

## 工作流 B — 修 Bug（快速修复）

**适用场景**：线上 Bug、测试失败、异常行为

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Step 1: 定位根因                                             │
│  ───────────                                                  │
│  👉 systematic-debugging                                      │
│                                                              │
│  严格流程：                                                    │
│  1. 复现 Bug（稳定重现步骤）                                    │
│  2. 最小化（排除无关因素）                                      │
│  3. 提假设（可能的原因列表）                                     │
│  4. 加仪器（加日志/断点验证假设）                                │
│  5. 修复                                                      │
│  6. 回归测试（确保修复且不引入新 Bug）                            │
│                                                              │
│  **铁律：找到根因前不要修复**                                    │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 2: TDD 修复                                             │
│  ───────────                                                  │
│  👉 test-driven-development                                   │
│                                                              │
│  1. RED: 写一个复现 Bug 的测试，看它失败                         │
│  2. GREEN: 写最小修复代码                                       │
│  3. REFACTOR: 清理                                            │
│                                                              │
│  **铁律：没有看到测试失败前，不写修复代码**                       │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 3: 验证完成                                              │
│  ───────────                                                  │
│  👉 verification-before-completion                            │
│                                                              │
│  - 跑全部测试，确认全绿                                         │
│  - 检查没有 console.log、as any、死代码                         │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**完整命令链**：
```
systematic-debugging → test-driven-development → verification-before-completion
```

## 工作流 C — 重构代码（安全改造）

**适用场景**：改善架构、解耦模块、消除技术债

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Step 1: 分析现状                                             │
│  ───────────                                                  │
│  👉 improve-codebase-architecture                             │
│                                                              │
│  做什么：                                                     │
│  - 扫描代码库发现"加深机会"（deepening opportunities）          │
│  - 检查 CONTEXT.md 中的领域语言是否与代码一致                   │
│  - 推荐模块边界改造方案                                         │
│                                                              │
│  这个技能会派出 3 个子 Agent 并行分析，                           │
│  每个给出不同的接口设计方案                                      │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 2: 确认重构范围                                           │
│  ───────────────                                               │
│  👉 grill-with-docs（限焦在重构区域）                            │
│                                                              │
│  不是为了设计新功能，而是为了：                                   │
│  - 验证 proposed 接口设计是否合理                                │
│  - 明确重构边界（什么改、什么不改）                               │
│  - 更新 CONTEXT.md 和 ADR                                      │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 3: TDD 改造                                              │
│  ───────────                                                  │
│  👉 test-driven-development                                   │
│                                                              │
│  注意：重构前先确保现有行为有测试覆盖                              │
│  没有测试的代码 → 先加 Characterization Test（记录当前行为）       │
│  然后重构，保证测试一直绿                                        │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 4: 收尾验证                                              │
│  ───────────                                                  │
│  👉 verification-before-completion                            │
│     finishing-a-development-branch                            │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**完整命令链**：
```
improve-codebase-architecture → grill-with-docs → test-driven-development
  → verification-before-completion → finishing-a-development-branch
```

## 工作流 D — 性能优化（专项审计）

**适用场景**：构建太慢、页面卡顿、包体积过大

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Step 1: 性能审计                                             │
│  ───────────                                                  │
│  👉 vue-vite-performance-audit                                │
│                                                              │
│  4 层检查：                                                    │
│  Layer 1: Vite 配置（是否用了正确的构建选项）                    │
│  Layer 2: 依赖分析（是否有冗余/过大的依赖）                      │
│  Layer 3: 运行时（组件渲染、响应式性能）                         │
│  Layer 4: 构建产物（chunk 拆分、Tree-shaking）                  │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 2: 查询现代 Web 方案                                      │
│  ────────────────                                             │
│  👉 modern-web-guidance                                       │
│                                                              │
│  针对审计发现的问题，查询最新的 Web API 最佳实践：                │
│  - 是否有新的 CSS 特性可以替代 JS 方案？                         │
│  - View Transitions API？                                      │
│  - content-visibility 懒渲染？                                 │
│  - 图片优化（fetchpriority、loading="lazy"）？                  │
│                                                              │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  Step 3: 架构改进（如需要）                                     │
│  ────────────────                                             │
│  👉 improve-codebase-architecture                             │
│                                                              │
│  如果审计发现模块耦合导致的性能问题                                 │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

**完整命令链**：
```
vue-vite-performance-audit → modern-web-guidance → improve-codebase-architecture
```

```
│  ─────────────────────                                       │

│                                                              │

│                                                              │
```

## 工作流 H — 小改动（轻量速攻）

**适用场景**：改个文案、修个小样式、加个简单逻辑

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  Step 1: 快速理解需求                                          │

│                                                              │
│  Step 2: 执行                                                  │
```

## 📊 快速选择矩阵

| 你遇到的情况 | 选哪个工作流 | 核心技能 | 预计耗时 |
|------------|:----------:|---------|:-------:|
| "我要做一个新功能" | **A** | grill-with-docs → prototype → to-prd → tdd | 中-长 |
| "我想加个小按钮" | **H** | brainstorming → 直接改 | 短 |
| "线上有个 Bug" | **B** | systematic-debugging → tdd | 中 |
| "这模块太乱了要重构" | **C** | improve-codebase-architecture → grill-with-docs → tdd | 长 |
| "页面加载太慢了" | **D** | vue-vite-performance-audit → modern-web-guidance | 中 |
| "这项目是什么结构？" | **E** | understand → understand-explain | 短-中 |

## 💡 关键技巧

### 1. 不要在一个会话里做所有事

```
❌ 错误：
同一个会话里既 grill 又 prototype 又写代码
→ 上下文窗口爆炸，模型变笨

✅ 正确：
grill 会话（Step 1）→ /handoff → prototype 会话（Step 2）
→ /handoff → grill 会话 → /to-prd
→ 新会话执行
```

### 2. Low Fidelity vs High Fidelity

```
Low Fidelity（可以 grill）：路由命名、数据模型关系、模块划分
High Fidelity（需要 prototype）：UI 布局、交互动效、复杂状态流转

遇到 High Fidelity → 不要强行在 grill 里讨论 → 切 prototype
```

### 3. 用 "Smart" 模型 grill，用 "Fast" 模型执行

```
Grilling 需要模型的参数知识（创意设计建议）→ 用强模型
执行实现有详细计划作为上下文 → 可以用轻量模型
```

### 4. 每次 grill 的决策都要保存

```
Grill 产出的 CONTEXT.md + ADR + PRD 是跨会话可复用的资产
下次进入项目时，/grill-with-docs 会自动读取这些文件
你不需要重新解释术语和设计决策
```
