# Big O 记法

Big O 记法是一种不使用时间来描述函数性能的方法。与其测量函数从开始到结束的时间，Big O 描述的是时间如何随着输入大小的增加而增长。它用于帮助理解程序在各种输入范围内的性能表现。

在这篇文章中，我将介绍 4 种常用的 Big O 记法类别：**常数**、**对数**、**线性**和**二次**。如果你现在对这些词感到陌生，别担心。我将在整篇文章中详细讨论它们，并可视化展示。

## 迭代

考虑这个计算 1 到 n 之和的 JavaScript 函数。

```javascript
function sum(n) {
  let total = 0;
  for (let i = 1; i <= n; i++) {
    total += i;
  }
  return total;
}
```

我要传入 10 亿，使用简写 `1e9`，这样它需要可观察到的时间来运行。按下下面的按钮来测量计算 1 到 10 亿之和需要多长时间。

在我的笔记本电脑上，这只需要不到 1 秒的时间。你得到的时间可能不同，每次运行时可能会有几十毫秒的变化。这是预期的。

通过获取开始时间和结束时间并找到差值来对代码进行计时，这种方法称为**墙上时钟时间**。

你认为 20 亿，`2e9`，会需要多长时间？

它需要大约两倍的时间。代码循环 `n` 次，每次执行一次加法。如果我们传入 30 亿，我们预计执行时间会增加三倍。

函数的输入与执行所需时间之间的关系称为其**时间复杂度**，而 **Big O 记法**是我们交流函数时间复杂度的方式。

玩弄下面的按钮。每个条将额外的 10 亿添加到我们传递给 `sum` 的 `n` 中，所以 1e9 按钮调用 `sum(1e9)`，2e9 按钮调用 `sum(2e9)`，以此类推。你应该看到 `2e9` 大约需要 `1e9` 的两倍时间，`3e9` 大约需要 `1e9` 的三倍时间。

**注意**：在某些浏览器上，你可能会遇到条形花费的时间比预期的长得多。我在发布之前无法弄清楚原因。如果某个条形看起来完全偏离你的预期，尝试重新运行它。

因为 `sum` 的墙上时钟时间与 `n` 以相同的速率增长，例如，`sum(20)` 将花费 `sum(10)` 两倍的时间，所以我们说 `sum` 是一个"线性"函数。它的 Big O 是 n，或者 **O(n)**。

为什么我们要使用这种语法：**O(n)**？什么是 O？为什么要用那些括号？

"O"代表"order"，是"order of growth"（增长阶数）的缩写。大声说出来就是："`sum` 函数的增长阶数是 n"。**O(n)** 是一种简洁的写法。这种记法是由德国数学家保罗·巴赫曼（Paul Bachmann）在 1894 年创建的。另外，"O"（字母）在某些字体中可能看起来像 0（数字）。它总是字母 O。

计算 1 到 n 之数字和的另一种方法是使用公式 `(n*(n+1))/2`。下面是使用 1 到 5 的数字的这个公式的结果。在每种情况下，结果应该与执行例如 `1+2+3+4+5` 对于 `n=5` 的结果相同。

-   `(1*2)/2 = 2/2 = 1`
-   `(2*3)/2 = 6/2 = 3`
-   `(3*4)/2 = 12/2 = 6`
-   `(4*5)/2 = 20/2 = 10`
-   `(5*6)/2 = 30/2 = 15`

如果 `sum` 使用这个公式而不是我们之前的循环，它会是什么样子：

```javascript
function sum(n) {
  return (n * (n + 1)) / 2;
}
```

你认为这个函数的墙上时钟时间如何随着 `n` 的增加而变化？接下来的两个示例相差 100 倍。

这个示例没有问题。这两个函数几乎不需要任何时间。计时的变化是由浏览器和计算机的不可预测性引起的，而不是 `sum` 函数。每次运行每个示例几次，你应该看到两个函数的墙上时钟时间都在相同值附近徘徊。

我们称这样的函数为**常数**或 **O(1)**，无论你给它什么输入，它的墙上时钟时间大致相同。

哇，所以我们将 `sum` 函数从 **O(n)** 改进到了 **O(1)**！它现在总是瞬间运行！

是的！但关键是记住 **O(1)** 并不*总是*意味着"瞬间"。它意味着**所需时间不会随输入大小而增加**。对于我们的新 `sum` 函数，它几乎是瞬间的。但是，根据它所做的事情，一个 **O(1)** 算法可能需要几分钟甚至几个小时。

对于某些输入，一个 **O(n)** 算法也可能比一个 **O(1)** 算法更快。然而，随着输入大小的增长，**O(1)** 算法最终会超越 **O(n)** 算法。使用下面的滑块查看一个示例。

在那张图中，**O(1)** 线始终在 20，那么我们为什么不说是 **O(20)** 呢？

Big O 记法的目的是描述输入和函数墙上时钟时间之间的关系。它不关心墙上时钟时间最终是多少，只关心它如何增长。

Big O 用尽可能小的项来描述增长。如果世界必须找出每个函数在括号内放置什么数字，并且这些数字相对于彼此是正确的，那会很快变得混乱。同样，线性函数总是 **O(n)** 而从不 **O(2n)** 或 **O(n + 1)**，因为 **O(n)** 是最小的线性项。

## 排序

让我们远离 `sum`，讨论一种不同的算法：**冒泡排序**。

冒泡排序背后的思想是你循环遍历输入数组，如果相邻的元素不按期望的顺序排列，就交换它们。你这样做直到能够完成一次通过数组而不进行任何交换。它被称为冒泡排序是因为数字"冒泡"到其正确位置的方式。

下面是该算法的 JavaScript 实现。我们要将这些数字按升序排序，所以期望的结果是 1、2、3、4、5。

你可以使用控件逐步执行此代码并观看数组被排序。向前一次前进 1 行，向后，每秒自动前进 1 行，并重置回开始。

```javascript
function bubbleSort(a) {
  while(true) {
    let swapped = false;
    for (let i = 0; i < a.length - 1; i++) {
      if (a[i] > a[i+1]) {
        [a[i], a[i+1]] = [a[i+1], a[i]];
        swapped = true;
      }
    }
    if (!swapped) break;
  }
  return a;
}
```

```javascript
bubbleSort([3, 2, 5, 4, 1]);
```

现在你已经看到算法在运行，你认为它的 Big O 是什么？

如果数组已经排序，算法循环一次，不进行交换，然后退出。这将是 **O(n)**。但是如果数组是任何其他顺序，我们需要循环不止一次。在最坏的情况下，数组是反向顺序的，我们将不得不循环 `n` 次以将最小元素从末尾移动到开头。

循环我们数组的 `n` 个元素 `n` 次导致 `n*n` 次操作，或 `n^2`。这意味着**冒泡排序是一个 O(n^2) 算法**。有时称为**二次**。

因为算法的性能通常不仅取决于输入的*大小*，还取决于其排列，所以**Big O 记法，除非另有说明，描述最坏情况场景**。相同的记法也可以用于最好和平均情况，但最坏情况是默认的。

下面是另一种可视化冒泡排序的方法。它从顶向下开始于数组 5、4、3、2、1。最终状态应该是从顶向下 1、2、3、4、5。每前进一步代表完整地通过数组一次，可能涉及多次交换。使用 Best（最好）、Worst（最坏）和 Rand（随机）来查看不同的初始配置。

如果你玩了几次 Rand，你会注意到你有时确实只得到几次迭代。尽管如此，冒泡排序仍然是 **O(n^2)**，因为在最坏情况下你将不得不遍历数组 `n` 次。

## 将这些知识付诸实践

在文章的前面部分，我描述了 **O(1)**、**O(log n)**、**O(n)** 和 **O(n^2)** 算法之间的区别。让我们看看你在编写代码时可能遇到的一些情况以及你可以做些什么来改进你的时间复杂度。

### 在列表中查找项目

这里有一个函数，用于检查列表是否包含特定项目。

```javascript
function contains(items, target) {
  for (const item of items) {
    if (item === target) {
      return true;
    }
  }
  return false;
}
```

如果你在同一个列表中查找项目很多次，你可能想考虑使用一种允许更快查找的数据结构，比如 **Set**。现代浏览器以一种为查找提供 **O(1)** 时间复杂度的方式实现 `Set`。

但是，不要这样做：

```javascript
function contains(items, target) {
  const itemSet = new Set(items);
  return itemSet.has(target);
}
```

*构建* `new Set(items)` 是一个 **O(n)** 操作！这是因为 `Set` 构造函数循环遍历数组中的所有项目以将它们添加到集合中。你需要权衡这种前期工作的成本与更快查找的潜在节省。

```javascript
const items = new Set(["apple", "banana", "cherry"]);
items.has("banana") // true, and O(1)!
```

### 使用索引循环数组

下面的代码包含我在生产代码中看到过几十次的常见错误。阅读一下，看看你是否能回答：

-   这个函数的 Big O 是什么？
-   我们如何改进它？

```javascript
function buildList(items) {
  const output = [];
  for (const item of items) {
    const index = items.indexOf(item);
    output.push(`Item ${index + 1}: ${item}`);
  }
  return output.join("\n");
}
```

问题是在循环内调用 `.indexOf`。`.indexOf` 函数是一个 **O(n)** 操作！它的工作是循环遍历数组，直到找到目标元素，如果没有找到则返回 `-1`。在循环内调用它使我们 `buildList` 函数的整体 Big O 变为 **O(n^2)**！

为了修复这个问题，我们可以使用索引进行循环。通过索引在数组中查找元素是 **O(1)**，所以函数的整体 Big O 减少到 **O(n)**。

```javascript
function buildList(items) {
  const output = [];
  for (let i = 0; i < items.length; i++) {
    output.push(`Item ${i + 1}: ${items[i]}`);
  }
  return output.join("\n");
}
```

你也可以使用 JavaScript 的 `items.forEach((item, index) => ...)` 数组方法或 `items.entries()` 实现相同的结果。

### 缓存中间结果

考虑这个计算数字的阶乘的函数。数学中的阶乘写成，例如，`5!` 表示 `5*4*3*2*1` 或 `3!` 表示 `3*2*1`。

```javascript
function factorial(n) {
  if (n === 0) {
    return 1;
  }
  return n * factorial(n - 1);
}
```

这个函数的时间复杂度是 **O(n)**，但大多数对这个函数的调用将重做我们之前做过的工作。调用 `factorial(4)` 然后 `factorial(5)` 将意味着 `factorial(4)` 被计算两次。

我们可以缓存每个计算的结果以避免这种冗余工作。

```javascript
const cache = new Map();
function factorial(n) {
  if (cache.has(n)) {
    return cache.get(n);
  }
  if (n === 0) {
    return 1;
  }
  const result = n * factorial(n - 1);
  cache.set(n, result);
  return result;
}
```

这利用了 `Map` 中查找的 **O(1)** 时间复杂度。它不会改变 `factorial` 函数的最坏情况时间复杂度，但它确实以增加内存使用为代价使平均情况更快。

当谈到性能时，记住你不能总是相信你在网上读到的内容。**总是在更改之前和之后测试你的代码**，以确保你确实在改进它！

## 结论

让我们回顾一下我们学到的内容：

-   Big O 记法描述了**函数输入与其墙上时钟时间之间的关系**。
-   从增长最慢到增长最快，我们看到了以下示例：
    -   **O(1)**，常数时间（最好！）
    -   **O(log n)**，对数时间
    -   **O(n)**，线性时间
    -   **O(n^2)**，二次时间
-   我们可以通过做出更好的算法选择和避免常见的陷阱来改进我们编写代码的时间复杂度。
