# 如何评估 npm 包 - 2026 年版

每次运行 `npm install` 时，你都在添加将在生产环境中执行的代码：这些代码由你从未见过的人编写，可以访问你的进程所能触及的任何内容。它可能触及你的文件系统，发起出站网络请求，读取环境变量，或者悄悄地泄露数据。实际上，你是将基础设施信任给了一个陌生人。

大多数开发者通过检查两个数字来管理这种风险：每周下载量和 GitHub 星数。这两个数字都不能告诉你任何关于包是否安全、是否得到维护，或者是否诚实说明其功能的有意义信息。（大多数 npm 包使用 GitHub。如果项目托管在其他地方，应用相同的原则。）

供应链攻击使情况变得更糟。event-stream、ua-parser-js、node-ipc、xz utils —— 模式是一致的：一个合法的、广泛使用的包被攻破，要么是通过维护者被社会工程攻击，要么是拼写劫持（typosquat），要么是埋藏在三层深的依赖项。npm 生态系统，以其小型可组合包和深度传递依赖树的文化，是一个特别有吸引力的目标。你可以做对一切，但仍然会通过一些你从未直接安装的东西受到攻击。

还有一种值得了解的新变体。AI 编程助手会臆造包名。它们自信地建议 `npm install some-plausible-sounding-package`，但那些包并不存在。攻击者会监视这些臆造并注册名称 —— 这种技术现在被称为 **slopsquatting** —— 所以当开发者在没有检查的情况下遵循建议时，他们会安装恶意软件。如果 LLM 建议了一个你从未听说过的包，请验证它是否存在、有真实的历史，并且在运行安装之前有来源证明（provenance）。

当你以代理模式运行 LLM 时，这种风险会增加。在许多设置中，包的选择和安装是连续发生的，没有人工检查点。如果你这样做，请依赖工具链中的强制执行（例如，阻止未知或未验证包的包装器或安装挂钩），而不是依赖"记住先检查"的提示文本。

以上所有并不意味着你应该停止使用开源包 —— 这会让你效率降低，而不会让你有意义地更安全。这意味着选择一个包值得不仅仅是看一眼星数。

本指南为你提供了一个在添加 npm 包之前对其进行评估的可重复流程。它需要 5 到 10 分钟。它不会保证安全性 —— 没有什么能保证 —— 但它将帮助你做出明智的决定，而不是乐观的决定。

## [0. 你真的需要这个包吗？](#0.-do-you-actually-need-this-package%3F)

在你审计任何东西之前，先问最简单的问题：这个依赖项应该存在于你的项目中吗？

许多 npm 事件变得严重不是因为一个包本身是灾难性的，而是因为一个微小的便利包被复制到数十个服务和前端应用中，直到无处不在。如果该包被攻破、遗弃或突然取消发布，你的影响范围就不再很小了。

### [要检查的内容：](#what-to-check%3A)

在脑海中做一个**移除测试**：如果这个包明天消失了，替换它会有多难？如果答案是"我们必须重构一半代码库"，请将其视为高风险并应用更严格的审查。

如果它是一个微小的实用程序，请询问你是否可以在自己的代码库中用几行代码实现相同的东西。为一个辅助函数引入依赖通常不值得长期风险。

检查包的依赖项足迹。零运行时依赖的包并不自动安全，但更少的依赖通常意味着更小的攻击面和更少的传递惊喜。在 npm 上，检查依赖项计数并扫描那些依赖项实际上是什么。

然后检查你计划在哪里使用它。在一个孤立的内部工具中使用的包与将在每个服务中导入的包具有不同的风险概况。

## [1. 它是否正在积极维护？](#1.-is-it-actively-maintained%3F)

未维护的包是随时间累积的负债。安全漏洞未修补。与较新 Node 版本的兼容性静默地破裂。API 冻结，而生态系统在前进，最终你被固定在某个东西的旧版本上，因为更新它将需要替换一个不再有人触及的包。

明显的信号是最近的提交，但仅靠提交频率会产生误导。一个包可能上周有一个提交，但它只是更新了一个 CI 操作。你要寻找的是作者是否仍然参与实际的软件。

### [要检查的内容：](#what-to-check%3A-1)

在 GitHub 仓库上，转到 **Issues** 标签。查看最旧的未解决问题。它们是否被确认？如果有人在 18 个月前报告了一个错误，而作者从未回复，这会告诉你当出现问题时维护关系是什么样的。

查看 **Commits** 标签。过滤掉机器人提交和 CI 噪音。人类最后一次对源代码进行有意义的更改是什么时候，而不仅仅是依赖项升级或工作流调整？

查看谁在做这项工作。如果几乎每个有意义的提交、发布和问题回复都来自一个人，你就会有**维护者集中风险**（"bus factor" —— 在项目陷入困境之前需要有多少人被公交车撞到）。这并不自动是坏事 —— 许多优秀的包由一个维护者运行 —— 但这意味着你的操作风险与一个人的可用性和精力绑定在一起。

在 **npm 页面**上，转到 Versions 标签。是否有可识别的发布节奏 —— 每月、每季度，无论什么 —— 或者是 2 年的空档后突然爆发出活动？长时间的空档后紧接着突然更新有时本身就是危险信号：账户确实会被接管。

检查 **CHANGELOG**。如果它只列出提交哈希，那几乎没用。一个说"Fixed: deduplication plugin now clones responses per waiter to prevent body-already-used errors"的变更日志是由关心你是否了解更改内容和原因的人写的变更日志。变更日志的质量是作者如何看待使用其软件的人的代理。

最后，如果包暴露了随时间变化的公共 API，请查找迁移指南。记录破坏性更改并提供升级路径的作者是一个考虑其决策的下游成本的作者。

## [2. 你能信任发布到 npm 的内容吗？](#2.-can-you-trust-what's-actually-published-to-npm%3F)

维护是关于未来的。本节是关于更直接的内容：npm 注册表上的内容是否实际上是作者打算发布的内容。

npm 注册表默认没有验证。当你安装一个包时，你信任你接收到的字节与你在 GitHub 上可以读取的源代码匹配。对于大多数包，大多数时间，这是真的。但将源代码链接到发布的机制 —— 存储为 CI 秘密的 `NPM_TOKEN`，有时只是开发人员笔记本电脑上的东西 —— 正是攻击者针对的那种凭证。

2018 年的 event-stream 攻击是最明显的例子：维护者将流行包的控制权交给了陌生人。陌生人发布了一个带有恶意依赖的版本。在任何人注意到之前，数百万项目受到影响。没有人黑进 GitHub。没有人破坏 npm 的基础设施。他们只是获得了凭证。

[**来源证明认证**](https://docs.npmjs.com/cli/v9/commands/npm-publish#provenance)是现代的答案。当包与来源证明一起发布时，npm 注册表会收到加密认证（由 GitHub 的 OIDC 基础设施签名），该认证将特定的包 tarball 链接到特定仓库中的特定提交，由特定的 GitHub Actions 工作流运行构建。你可以验证它。认证是公开的。如果有人发布声称来自特定仓库的包，但认证不匹配，那是可检测的。

### [要检查的内容：](#what-to-check%3A-2)

转到 `npmjs.com/package/<package-name>`。在版本名称旁边，如果包与来源证明一起发布，应该有一个绿色的"Provenance"徽章：

![来源证明徽章](badge.png)

点击它，然后点击"View more details"链接。它将带你到页面底部：

![来源证明详情](provenance.png)

它显示仓库 URL、提交 SHA 和发布包的 GitHub Actions 工作流运行。你可以点击所有这些东西来验证它们存在且有道理。如果徽章在那里并且细节检查出来，你可以合理地确信你安装的包是作者打算发布的内容。

如果它不在那里，包是在没有来源证明的情况下发布的 —— 这并不自动可疑（大多数包早于该功能），但这意味着你无法验证源到发布的链路。

你也可以从命令行检查。在任何安装了包的项目中，运行：

```sh
npm audit signatures
```

这验证所有已安装包的加密签名，并报告哪些包具有有效的来源证明认证。

如果你想查看包是如何发布的，请在仓库中找到 `.github/workflows/` 目录并打开发布工作流。查找三件事：

-   `npm publish --provenance` —— 这是生成认证的内容
-   作业权限中的 `id-token: write` —— 这允许与 npm 进行 OIDC 令牌交换
-   缺少作为秘密的 `NPM_TOKEN` —— 如果工作流使用可信发布（OIDC），则没有长期令牌可被窃取

最后，查看工作流文件中如何引用 GitHub Actions。引用为 `uses: actions/checkout@v4` 或 `uses: actions/setup-node@main` 的操作固定到可变标签 —— 操作作者可以随时更改该标签指向的内容，你的工作流将静默地开始运行不同的代码。固定到完整提交 SHA（`uses: actions/checkout@de0fac2e4500dabe0009e67214ff5f5447ce83dd`）的操作无法更改，除非更新工作流文件本身中的引用。这是一件小事，但它关闭了一个真正的攻击面。

还有一件事会在你审查任何内容之前在你的机器上运行代码：安装脚本。`package.json` 中的 `preinstall`、`install` 和 `postinstall` 钩子在你运行 `npm install` 时立即执行。大多数合法的包不需要它们 —— 编译 C++ 绑定的原生插件是主要的真正用例。如果你在包中看到安装脚本但没有明显的理由，这值得在继续之前了解。

## [3. CI 管道是真实的还是装饰性的？](#3.-is-the-ci-pipeline-real-or-decorative%3F)

README 中的绿色徽章很容易伪造，或者更确切地说，很容易获得而不意味着什么。仓库可以有一个在单个快乐路径上运行三个测试并报告 100% 通过率的持续集成。该徽章在技术上是准确的，但完全无用。

你想知道的是 CI 管道是否真正保护代码库：它是否会在发布之前捕获回归、类型错误或损坏的边缘情况。

### [要检查的内容：](#what-to-check%3A-3)

转到 GitHub 上的 **Actions** 标签。查看工作流运行。它们是否在 `pull_request` 事件上触发，而不仅仅是在推送到 `main` 时？仅在合并后运行的管道不保护任何东西，它只是产生已经发生的事情的记录。

打开一个最近的合并拉取请求。CI 在其上运行了吗？有人必须等待它通过才能合并吗？如果 PR 在打开 30 秒后合并，没有 CI 运行，那么管道是装饰性的。

查找测试配置文件：`vitest.config.js`、`jest.config.js` 或等效文件。查找覆盖率阈值。像这样的东西：

```js
thresholds: {
  lines: 90,
  functions: 90,
  branches: 90,
}
```

如果配置了阈值，当覆盖率低于它们时，CI 管道将失败。如果未配置，可能会报告覆盖率但未强制执行，作者可以删除一半的测试，构建仍然会通过。

还要查看测试实际涵盖的内容。镜像 `src/` 结构的 `test/` 目录，或与源文件共存的单元测试，是一个好迹象。只有少量冒烟测试的 `index.test.ts` 文件完全是另一回事。你无法详细审计测试，但你可以获得一种感觉，即作者是否认真对待它们。

## [4. 代码质量是否可见？](#4.-is-the-code-quality-visible%3F)

你不会审计你考虑的每个包的整个代码库。那是不现实的。但你可以快速查看与代码质量相关的信号，作者做或不做的可见的事情，往往可以预测包是健壮还是脆弱。

### [要检查的内容：](#what-to-check%3A-4)

查看 linting 配置：`eslint.config.js`、`.eslintrc` 或等效文件。它存在吗？它是否非平凡？空白或近乎空的 linting 配置表明作者没有强制执行一致性或自动捕获明显的错误。Linting 是基本赌注；它的缺失是一个信号。

检查包是否提供格式良好的包。查看 `package.json` 中的 `exports` 字段。现代包应指定命名导出条件：如果它支持两个模块系统，则为 `import` 和 `require`，或者如果它故意仅限 ESM，则仅为 `import`，理想情况下如果是它提供 TypeScript 声明，则为 `types`。仅设置 `main` 而根本没有 `exports` 字段的包是在这成为标准实践之前编写的。这不是取消资格的，但它告诉你一些关于作者实践当前状态的信息。

查看 `package.json` 中的 `prepublishOnly` 脚本：

```json
"prepublishOnly": "npm run build && npm run test:ci"
```

这防止作者意外发布损坏的构建或跳过测试的构建。它不防止恶意发布，但它确实告诉你作者已经考虑了意外发布。

**特别是对于 TypeScript 包：**打开 `tsconfig.json` 并检查是否设置了 `strict: true`。严格模式启用空检查、严格函数类型和 no-implicit-any —— 一类在编译时而不是在生产中捕获的错误。关闭它的作者已决定其中一些错误是可以接受的。

还要在仓库中搜索 `any` 和 `@ts-ignore`。在真正尴尬的互操作情况下使用几次是正常的。在源代码中分散几十个意味着 TypeScript 类型在很大程度上是装饰性的：包有 `.ts` 扩展名但没有类型安全。

## [5. 当出现问题时会发生什么？](#5.-what-happens-when-something-goes-wrong%3F)

每个非平凡的包最终都会有安全漏洞。问题是作者在它发生时如何处理它。他们是否快速响应？他们是否负责任地披露 —— 先私下，然后公开发布修复？他们是否记录发生的事情，以便你可以评估你的版本是否受到影响？

### [要检查的内容：](#what-to-check%3A-5)

在仓库根目录中查找 `SECURITY.md` 文件，或转到 `github.com/<owner>/<repo>/security/policy`。这应该告诉你如何私下报告漏洞、联系方式、响应时间表，以及一些表明作者认真对待披露的迹象。它的存在并不意味着包不安全，但这意味着如果你发现问题，你没有明确的路径可以在不意外公开披露的情况下报告它。

转到 GitHub 上的 **Security** 标签并查看 Advisories 部分。有什么内容发布吗？如果是，它是如何处理的 —— 披露是否协调，发布时是否有修复可用，是否清楚记录了受影响的版本范围？处理良好的历史建议实际上是一个积极的信号；这意味着作者知道负责任的披露过程是什么样的并遵循了它。

在 `osv.dev` 或 `snyk.io` 上检查包中的已知漏洞。这些聚合 CVE 和 GitHub 安全建议。如果包有已知未修补的漏洞，这是你在安装之前需要知道的事情。

[Socket.dev](https://socket.dev) 比 CVE 数据库更进一步。与其等待漏洞被报告和编目，它进行行为分析：这个包是否访问网络？它是否以意外方式触及文件系统？它是否包含混淆代码？它是否有安装脚本？它还有一个 GitHub 应用程序，它对拉取请求运行此分析，并在合并之前标记新依赖项。对于你认真考虑的包，值得快速检查。

如果你在高保证环境中运行，[Socket Firewall](https://socket.dev/features/firewall) 也值得了解。它不仅在审查时间发出警告，还在安装时间强制执行包策略，并可以完全阻止已知恶意包进入你的环境。这对于业余项目通常是大材小用，但对于受监管或安全敏感的系统，它可以是一个强大的额外控制。

最后，查看过去的安全问题得到解决的速度。几天内修复的关键漏洞与敞开三个月的漏洞是不同的。安全问题上的响应时间是作者认真对待维护公共包责任的最明确信号之一。

## [检查清单](#the-checklist)

并非所有这些检查都具有同等的权重。如果你只有时间做三件事：

1.  **你真的需要它吗？**如果你可以快速替换它或完全避免它，你就是在消除风险而不是管理风险。
2.  **它有来源证明吗？**这是已发布包是作者打算发布内容的最明确信号。
3.  **它有无法解释的安装脚本吗？**代码在你运行 `npm install` 的那一刻就在你的机器上运行。

如果包是生产关键的，添加第四项检查：**维护者是否响应？**如果出现问题，你想知道能够修复它的人实际上会修复它。

当赌注更高时，其他一切都值得检查。

### [安全关键信号](#security-critical-signals)

这些影响包是否可以安全安装以及你安装的内容是否是作者发布的内容。

| 信号 | 在哪里检查 | 绿色 | 红色 |
|------|------------|------|------|
| 来源证明 | npmjs.com 版本页面 | "Provenance" 部分存在且与仓库匹配 | 无来源证明，或版本来自未知源 |
| 可信发布 | `.github/workflows/publish.yml` | OIDC + `--provenance`，无 `NPM_TOKEN` | `NPM_TOKEN` 秘密，手动发布步骤 |
| 安装脚本 | `package.json` scripts 字段 | 不存在，或明显的原生插件原因 | `preinstall`/`postinstall` 没有明确的理由 |
| 固定 CI 操作 | `.github/workflows/*.yml` | SHA 固定的第三方操作 | `@v3`、`@latest`、`@main` |

### [运营成熟度信号](#operational-maturity-signals)

这些告诉你作者如何认真对待维护、质量以及依赖其包的长期成本。

| 信号 | 在哪里检查 | 绿色 | 红色 |
|------|------------|------|------|
| 积极维护 | GitHub 提交 + 未解决问题 | 最近 3 个月内有提交，问题已确认 | 最后一次提交 2 年以上，陈旧问题被忽略 |
| 依赖项足迹 | npm 包页面 + lockfile 树 | 对于包范围依赖项很少，没有令人惊讶的传递树 | 对于微不足道的实用程序有大量传递树 |
| 维护者集中 | 提交、发布、问题/PR 响应 | 工作分布在多个活跃维护者之间 | 一个维护者处理几乎所有代码、发布和支持 |
| 强制覆盖率 | vitest/jest 配置 | 阈值配置为 80%+ | 无阈值，或覆盖率徽章从未更改 |
| 安全策略 | `SECURITY.md` 或 Security 标签 | 清晰的披露流程、联系方式 | 缺失，或只是没有联系方式的一般模板 |

在安全关键信号上干净但在运营成熟度上弱的包是经过计算的风险。未能通过安全关键信号的包是不同类别的问题。

## [风险不是二元的](#risk-is-not-binary)

这一切都不是通过/失败测试。这是一个风险评估，风险取决于上下文。

在 UI 中格式化日期的实用程序与位于你的服务和政府 API 之间的 HTTP 客户端具有不同的风险概况。每周下载 5000 万次的依赖项比下载 500 次的依赖项有更多眼睛关注。来自拥有专门安全团队的组织的包与由单个开发人员在业余时间维护的包是不同的。这些都不是取消资格的条件 —— 生态系统中一些最可靠的包由个人维护 —— 但它们影响你应该应用多少审查。

目标不是在每个维度上找到得满分的包。目标是了解当你将其添加到项目时你接受什么，并刻意做出该决定，而不是默认做出。
