## 一、Clerk 是什么

### 1.1 概述

**Clerk** 是一个开发者优先的**托管式身份认证和用户管理平台**，由 Colin Sidoti 和 Braden Sidoti 于 2019 年创立。它不仅仅是"认证工具"，更是覆盖用户管理全生命周期的一体化服务。

> 核心定位：帮你从零到一快速集成用户系统，无需自建登录、注册、权限、会话管理、组织管理等基础设施。

2026 年的现状：Clerk 已被众多 Next.js/React 团队视为"默认选择"。Vercel CEO Guillermo Rauch 评价："其 `<SignIn />` 组件的内置最佳实践，团队自己实现需要数月。"

### 1.2 核心优势

| 优势 | 说明 |
|------|------|
| **🚀 预构建 UI 组件** | `<SignIn />`、`<SignUp />`、`<UserButton />`、`<UserProfile />`、`<OrganizationSwitcher />` 等开箱即用 |
| **🧩 多认证策略** | 密码、无密码、Magic Link、OAuth、MFA、Passkeys、SSO/SAML |
| **🏢 原生多租户** | 组织管理、自定义角色、邀请流程、域名自动加入，适合 B2B SaaS |
| **🔒 企业级安全** | SOC 2、HIPAA（BAA）、混合认证模型、欺诈检测 |
| **⚡ 极佳 DX** | Clerk CLI 一行命令初始化，Framework-aware 自动配置 |
| **🌐 广泛框架支持** | Next.js、React、Vue、Nuxt、Expo、Express、Fastify、Go、Python 等 |
| **💰 慷慨免费额** | 0-10,000 MAU 免费，之后按量计费 |

## 二、Better Auth 邮箱密码注册登录流程

### 2.1 概述

**Better Auth** 是一个**开源、TypeScript-first、框架无关**的认证库。它的核心设计哲学是"你的认证数据活在你的私有数据库里"——不像 Clerk/Auth0 那样把用户数据托管在第三方服务，Better Auth 直接在你的数据库里建表存数据。

### 2.2 数据库表结构

Better Auth 在你的私有数据库中管理以下 4 张表：

#### user 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `text` PK | 唯一标识 |
| `name` | `text` NOT NULL | 用户显示名 |
| `email` | `text` UNIQUE NOT NULL | 邮箱 |
| `emailVerified` | `boolean` | 是否已验证 |
| `image` | `text?` | 头像 URL |
| `createdAt` | `timestamp` | 创建时间 |
| `updatedAt` | `timestamp` | 更新时间 |

#### account 表（密码存这里！）

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `text` PK | 唯一标识 |
| `userId` | `text` FK → user.id | 关联用户 |
| `accountId` | `text` NOT NULL | 邮箱密码模式下 = email |
| `providerId` | `text` NOT NULL | 邮箱密码模式固定为 `"credential"`；社交登录=平台名 |
| `password` | `text?` | **加密后的密码哈希（scrypt）** |
| `accessToken` / `refreshToken` | `text?` | OAuth 令牌（社交登录用） |

> ⚠️ **密码不存储在 user 表，而是存储在 account 表**中。这为后续接入 OAuth（Google/GitHub/微信）提供了统一的数据模型。

#### session 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | `text` PK | 唯一标识 |
| `userId` | `text` FK → user.id | 关联用户 |
| `token` | `text` UNIQUE | 会话令牌（也是 Cookie 值） |
| `expiresAt` | `timestamp` | 过期时间（默认 7 天） |
| `ipAddress` / `userAgent` | `text?` | 设备信息 |

#### verification 表

用于邮箱验证、密码重置等临时令牌存储。

### 2.3 注册（Sign Up）完整流程

```
POST /sign-up/email
```

**Step 1 - 参数校验**
```typescript
signUpEmailBodySchema.parse({ name, email, password })
// 校验：name(required) + email(valid) + password(8-128 chars)
```

**Step 2 - 邮箱查重**
```sql
SELECT id FROM user WHERE email = 'zhangsan@example.com'
-- 已存在 → HTTP 422
```

**Step 3 - 密码哈希**
```typescript
// 默认 scrypt（Node.js 原生支持，OWASP 推荐）
const hashedPassword = await scrypt.hash(password, { N: 16384, r: 8, p: 1, keylen: 64 })
// 可自定义为 Argon2id
```

**Step 4 - 插入 user 表**
```sql
INSERT INTO user (id, name, email, emailVerified, createdAt, updatedAt)
VALUES ('usr_abc123', '张三', 'zhangsan@example.com', false, NOW(), NOW())
```

**Step 5 - 插入 account 表（存密码哈希）**
```sql
INSERT INTO account (id, userId, accountId, providerId, password, createdAt, updatedAt)
VALUES ('acc_xyz789', 'usr_abc123', 'zhangsan@example.com', 'credential', '<scrypt哈希>', NOW(), NOW())
```

**Step 6 - 创建 Session**
```typescript
const sessionToken = crypto.randomUUID()
const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)  // 7天
```

```sql
INSERT INTO session (id, userId, token, expiresAt, createdAt, updatedAt)
VALUES ('ses_456', 'usr_abc123', '<sessionToken>', '<7天后>', NOW(), NOW())
```

**Step 7 - 设置 Cookie**
```http
Set-Cookie: better-auth-session=<sessionToken>; HttpOnly; SameSite=Lax; Path=/; Max-Age=604800
```

### 2.4 登录（Sign In）完整流程

```
POST /sign-in/email
```

**Step 1 - 查 user 表**
```sql
SELECT * FROM user WHERE email = 'zhangsan@example.com'
-- 没找到 → 统一返回 "Invalid email or password"（防枚举）
```

**Step 2 - 查 account 表拿密码哈希**
```sql
SELECT password FROM account WHERE userId = 'usr_abc123' AND providerId = 'credential'
```

**Step 3 - scrypt 验证密码**
```typescript
const isValid = await scrypt.verify(hashedPassword, inputPassword)
```

**Step 4 - 创建新 Session + Set-Cookie**
```sql
INSERT INTO session (...token, userId, expiresAt...) VALUES (...)
-- → 设置 HttpOnly Cookie
```

## 三、开发者需要介入多少？

Better Auth 的设计理念是"声明式配置，框架自动处理"。开发者只需要做**一次性设置**，之后注册/登录的整个后端流程都由它自动完成。

### 3.1 你需要做的事（一次性设置）

#### ① auth.ts 配置文件（约 10 行）

```typescript
import { betterAuth } from "better-auth"
import { drizzleAdapter } from "better-auth/adapters/drizzle"
import { db } from "@/db"

export const auth = betterAuth({
  database: drizzleAdapter(db, { provider: "pg" }),
  emailAndPassword: { enabled: true },          // 一行开启邮箱密码认证
})
```

#### ② 挂载一个路由处理器

以 Next.js 为例，创建 `app/api/auth/[...all]/route.ts`：

```typescript
import { auth } from "@/lib/auth"
import { toNextJsHandler } from "better-auth/next-js"
export const { POST, GET } = toNextJsHandler(auth)
```

> 这一行就是一个**万能处理器**——Better Auth 会自动注册 sign-up、sign-in、sign-out、session 等所有 API 端点。

#### ③ 一行 CLI 建表

```bash
npx auth@latest generate   # 生成 ORM schema
npx auth@latest migrate    # 自动创建 4 张表
```

#### ④ 前端创建客户端

```typescript
import { createAuthClient } from "better-auth/react"
export const authClient = createAuthClient()
```

### 3.2 不需要你做的事

| 你以为要写的 | 实际 Better Auth 自动生成了 |
|------------|--------------------------|
| `POST /api/auth/sign-up` | ✅ 自动注册，含参数校验 + 邮箱去重 + 密码哈希 + 写表 |
| `POST /api/auth/sign-in` | ✅ 自动验证密码、查 account 表、创建 session |
| `POST /api/auth/sign-out` | ✅ 自动销毁 session、清除 Cookie |
| `GET /api/auth/session` | ✅ 自动验证 Cookie、查 session 表、返回用户信息 |
| 密码哈希 | ✅ 默认 scrypt，可换 Argon2id |
| Cookie 安全 | ✅ HttpOnly + SameSite=Lax + Secure |
| 暴力破解防护 | ✅ 内置 rate limiter |
| 邮箱枚举防护 | ✅ 统一错误信息 |

### 3.3 运行时数据流总览

```
【你的视角】                              【Better Auth 内部】
──────────────────────────────────────────────────────────────────
authClient.signUp.email({          POST /api/auth/sign-up/email
  name, email, password             → 参数校验（自动）
})                                  → 邮箱查重（自动：SELECT user）
                                    → 密码 scrypt 加密（自动）
                                    → INSERT user（自动）
                                    → INSERT account 存哈希（自动）
                                    → INSERT session（自动）
                                    → Set-Cookie（自动）
                                    → 返回响应（自动）
```

## 四、社交登录的数据存储与国内支持

### 4.1 社交登录数据存在哪？

社交登录与邮箱密码登录**共享同一套表**，只是 account 表的 `providerId` 不同。

**示例：同一用户绑定了邮箱、微信、GitHub**

```sql
-- user 表（一条记录）
id: "usr_abc123",  name: "张三",  email: "zhangsan@example.com"

-- account 表（三条记录，通过 userId 关联）
id: "acc_001",  userId: "usr_abc123",  providerId: "credential",  password: "<scrypt哈希>"
id: "acc_002",  userId: "usr_abc123",  providerId: "wechat",      accessToken: "<微信token>"
id: "acc_003",  userId: "usr_abc123",  providerId: "github",      accessToken: "<GitHub token>"

-- session 表（不管哪种方式登录成功，都创建会话）
id: "ses_001",  userId: "usr_abc123",  token: "...",  expiresAt: "..."
```

**关键设计**：
- `providerId = "credential"` → 邮箱密码登录
- `providerId = "wechat" / "google" / "github"` → 社交登录
- 同一个 `user` 可关联多个 `account`，实现多方式登录
- **session 表完全不变**，登录成功后统一创建会话

### 4.2 Better Auth 原生支持的社交提供商

从官方文档目录看，**原生内置** 30+ 社交提供商：

| 分类 | 提供商 |
|------|--------|
| **国际主流** | Google, GitHub, Apple, Microsoft, Facebook, Twitter/X, Discord, LinkedIn, Slack, GitLab, Spotify, Reddit, Twitch |
| **亚洲** | **WeChat（微信）**, Kakao（韩国）, Naver（韩国）, LINE（日本） |
| **企业工具** | Atlassian, Figma, Notion, Linear, Zoom, Hugging Face, Salesforce |
| **其他** | PayPal, TikTok, VK（俄罗斯）, Vercel, Dropbox, Spotify |

### 4.3 国内社交登录支持情况

| 国内平台 | 是否原生支持 | 接入方式 |
|---------|------------|---------|
| **微信 (WeChat)** | ✅ **是** | `socialProviders.wechat` 直接配置 |
| 微博 (Weibo) | ❌ 否 | Generic OAuth 插件手动配置 |
| 支付宝 (Alipay) | ❌ 否 | Generic OAuth 插件手动配置 |
| 钉钉 (DingTalk) | ❌ 否 | Generic OAuth 插件手动配置 |
| QQ | ❌ 否 | Generic OAuth 插件手动配置 |

**微信原生配置示例：**

```typescript
// auth.ts
export const auth = betterAuth({
  socialProviders: {
    wechat: {
      clientId: process.env.WECHAT_CLIENT_ID!,
      clientSecret: process.env.WECHAT_CLIENT_SECRET!,
      lang: "cn",
      scope: ["snsapi_login"],
    },
  },
})

// 客户端
await authClient.signIn.social({ provider: "wechat" })
```

> 微信支持**网站应用扫码登录**，需在[微信开放平台](https://open.weixin.qq.com/)注册应用获取 AppID。

**其他国内平台通过 Generic OAuth 接入（以微博为例）：**

```typescript
import { betterAuth } from "better-auth"
import { genericOAuth } from "better-auth/plugins"

export const auth = betterAuth({
  plugins: [
    genericOAuth({
      config: [{
        providerId: "weibo",
        clientId: process.env.WEIBO_CLIENT_ID!,
        clientSecret: process.env.WEIBO_CLIENT_SECRET!,
        authorizationUrl: "https://api.weibo.com/oauth2/authorize",
        tokenUrl: "https://api.weibo.com/oauth2/access_token",
        userInfoUrl: "https://api.weibo.com/2/users/show.json",
        scopes: ["email"],
      }],
    }),
  ],
})
```

> 任何实现 **OAuth 2.0 或 OpenID Connect (OIDC)** 协议的第三方平台都可以通过 Generic OAuth 插件接入。

## 五、跨语言后端如何验证用户身份

当 Better Auth（Node.js/TypeScript）处理认证，而业务接口用 Java 等其他语言开发时，有以下三种方案确保接口调用有效。

### 5.1 方案一：共享数据库验证（推荐）

**原理**：Java 后端直接读取同一份 `session` 表，验证 session token。

**架构图**

```
浏览器 ──Cookie──→ Node.js 认证服务 (Better Auth)
             │
             └──→ Java 业务服务 ──→ 共享数据库
                       │              user | account | session 表
                       ↓
                   处理业务（获取 userId）
```

**Java 实现**

```java
@RestController
public class OrderController {

    @GetMapping("/api/orders")
    public List<Order> getOrders(HttpServletRequest request) {
        // 1. 从 Cookie 取出 session token
        String sessionToken = extractCookie(request, "better-auth-session");
        if (sessionToken == null) throw new UnauthorizedException();

        // 2. 查共享数据库的 session 表
        Session session = sessionRepository.findByToken(sessionToken);
        if (session == null || session.getExpiresAt().isBefore(Instant.now())) {
            throw new UnauthorizedException("会话已过期");
        }

        // 3. 拿到 userId 处理业务
        return orderService.findByUserId(session.getUserId());
    }
}
```

**✅ 优点**：实时性强、实现简单
**⚠️ 注意**：需要共享数据库访问

---

### 5.2 方案二：JWT 无状态验证（性能最优）

**原理**：Better Auth 的 Cookie Cache 支持 JWT 策略，签发的 session cookie 本身就是 JWT。Java 端用相同密钥验证签名即可，无需查数据库。

**Better Auth 配置**

```typescript
export const auth = betterAuth({
  session: {
    cookieCache: {
      enabled: true,
      maxAge: 5 * 60,          // 缓存 5 分钟
      strategy: "jwt",         // ← 使用 JWT 策略
    },
  },
})
```

此时 `better-auth-session` Cookie 就是一个标准 JWT，包含：
```json
{ "sub": "usr_abc123", "email": "zhangsan@example.com", "iat": ..., "exp": ... }
```

**Java 验证**

```java
// 需要 jjwt 或 java-jwt 依赖
@Component
public class AuthValidator {
    // 必须与 BETTER_AUTH_SECRET 相同
    private final SecretKey signingKey = Keys.hmacShaKeyFor(
        System.getenv("BETTER_AUTH_SECRET").getBytes()
    );

    public Claims validateSession(String token) {
        return Jwts.parserBuilder()
            .setSigningKey(signingKey)
            .build()
            .parseClaimsJws(token)
            .getBody();
    }
}

@RestController
public class OrderController {
    @GetMapping("/api/orders")
    public List<Order> getOrders(
            @CookieValue("better-auth-session") String token) {
        Claims claims = authValidator.validateSession(token);
        String userId = claims.getSubject();
        return orderService.findByUserId(userId);
    }
}
```

**密钥共享**：Node.js 和 Java 共用同一个 `BETTER_AUTH_SECRET` 环境变量。

**✅ 优点**：零数据库查询、完全无状态
**⚠️ 注意**：JWT 缓存期内 session 撤销不会立即生效（可通过 `maxAge: 60` 缩短窗口）

---

### 5.3 方案三：API 网关模式（架构最干净）

**原理**：在 Node.js 层添加网关，统一验证 session 后将用户信息通过 Header 透传给 Java 后端。

**架构图**

```
浏览器 ──Cookie──→ Node.js 网关（验证 session）
                       │
                       │ 添加 Header: X-User-Id, X-User-Email
                       │
                       ▼
                  Java 业务服务（信任 Header，零认证逻辑）
```

**Node.js 网关**

```typescript
import { auth } from "./auth"

// 网关中间件
app.use("/api/business/*", async (req, res, next) => {
  const session = await auth.api.getSession({ headers: req.headers })
  if (!session) return res.status(401).json({ error: "未登录" })

  // 将用户信息写入 Header，转发给 Java
  req.headers["X-User-Id"] = session.user.id
  req.headers["X-User-Email"] = session.user.email
  next()
})
```

**Java 端**

```java
@RestController
public class OrderController {
    @GetMapping("/api/orders")
    public List<Order> getOrders(
            @RequestHeader("X-User-Id") String userId) {
        return orderService.findByUserId(userId);
    }
}
```

**✅ 优点**：Java 端零认证逻辑、集中管理
**⚠️ 注意**：网关和 Java 服务之间必须内网通信，防止绕过

---

### 5.4 方案对比

| 维度 | 共享数据库 | JWT 无状态 | API 网关 |
|------|-----------|-----------|---------|
| **实现复杂度** | ⭐⭐ 低 | ⭐ 极低 | ⭐⭐⭐ 中 |
| **性能** | 中（查 DB） | **最高**（零 IO） | 高（无 DB 但多一跳） |
| **实时撤销** | ✅ 即时 | ❌ 有缓存窗口 | ✅ 即时 |
| **Java 端依赖** | JDBC/JPA | JWT 库 | 无（只读 Header） |
| **架构耦合** | 中（共享 Schema） | 低（只共享密钥） | **最低** |
| **推荐场景** | 中小项目 | 微服务高并发 | 大型微服务 |

### 5.5 生产环境推荐组合

**方案二 + 方案三混合**：

```
                  ┌──────────────┐
                  │  Redis/DB    │
                  │  session 表   │
                  └──────┬───────┘
                         │
浏览器 → Node.js 网关 → Java 业务服务
           │                  │
           │ JWT 验证          │ 信任 X-User-Id Header
           │ 或降级查 DB       │
           │                  │
           │ 转发用户信息到 Header
```

- **普通接口**：JWT 验证（高性能）
- **关键操作**（支付、改密）：降级查 DB（实时性）

## 六、方案对比总结

### 6.1 Clerk vs Better Auth

| 维度 | Clerk | Better Auth |
|------|-------|-------------|
| **性质** | 托管 SaaS 服务 | 开源库，自托管 |
| **数据存储** | Clerk 的托管服务 | **你的私有数据库** |
| **定价** | 0-10K MAU 免费，之后按量 | MIT 协议，完全免费 |
| **开发者介入** | CLI 初始化 + 组件 | 配置 + 建表 + 挂载路由 |
| **社交登录** | 内置 30+ 平台 | 内置 30+ + Generic OAuth |
| **国内社交** | 微信等 | 微信原生 + 其余通过通用插件 |
| **跨语言集成** | JWT 验证（需 Clerk API） | 共享 DB / JWT / 网关三选一 |
| **UI 组件** | 丰富、美观、可定制 | 无内置 UI（需自建或使用第三方） |
| **适合** | 快速上线，不想操心认证 | 数据自控，需要私有部署 |
