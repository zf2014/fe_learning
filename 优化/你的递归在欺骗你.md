# 你的递归在欺骗你

> 原文: [Your Recursion Is Lying to You](https://blog.gaborkoos.com/posts/2026-05-09-Your-Recursion-Is-Lying-to-You/)
> 作者: Gabor Koos | 2026年5月9日

*入选 [Node Weekly #624](https://nodeweekly.com/issues/624)* 和 [Javascript Weekly - 2026-06-02](https://javascriptweekly.com/issues/788)*

[递归](https://developer.mozilla.org/en-US/docs/Glossary/Recursion) 是开发者很早就学到的概念之一，并且会信任很多年。如果递归步骤简单且基本情况正确，代码看起来既干净又安全。

它的优雅是有道理的：许多问题天然是递归的，而且代码往往与我们口头解释逻辑的方式一致。对于树遍历、嵌套结构和分治模式，递归比显式循环更容易阅读。

问题在于物理限制。即使基本情况正确、逻辑无误，每次递归调用仍然会消耗栈空间。当深度达到一定程度时，你就会遇到栈溢出而崩溃。

如果你读过[你的防抖在欺骗你](https://blog.gaborkoos.com/posts/2026-03-28-Your-Debounce-Is-Lying-to-You/)和[你的节流在欺骗你](https://blog.gaborkoos.com/posts/2026-03-31-Your-Throttling-Is-Lying-to-You/)，这篇则是同一模式的递归版本：优雅的抽象，隐藏的操作边缘状况。

---

## 问题设定：递归撞上南墙

你可以直接在浏览器控制台中运行以下所有代码。让我们从一个简单的例子开始：递归计算从 1 到 n 的所有整数之和。

```js
function sum(n) {
  if (n === 0) return 0;
  return n + sum(n - 1);
}

sum(10); // 55
```

现在输入一个大数：

```js
sum(100000); // 在大多数 JS 运行时中抛出 RangeError 或 InternalError: too much recursion
```

发生了什么？函数逻辑是正确的，但是每次调用 `sum` 都会留在栈上，直到它下面的调用返回。在深度 100,000 时，运行时耗尽栈空间并抛出异常。这与结果是否正确无关，纯粹是运行时能同时容纳的嵌套帧数量的物理限制。

## 尾递归救援故事

通常的下一步是[尾调用优化](https://en.wikipedia.org/wiki/Tail_call)（TCO）。思路很简单：让递归调用成为函数执行的最后一步，这样运行时可以复用同一个栈帧，而不是压入一个新帧。

注意，`sum` **不是**尾递归的——尽管递归调用出现在最后一行。在 `sum(n - 1)` 返回之后，仍然有待处理的工作：结果必须加上 `n`。**只有当返回值被立即转发、之后没有任何待处理计算时，调用才处于尾位置**。

尾递归版本将待处理状态移入累加器：

```js
function sumTR(n, acc = 0) {
  if (n === 0) return acc;
  return sumTR(n - 1, acc + n);
}

sumTR(10); // 55
```

这里 `sumTR(...)` 是最后发生的事情——没有待处理的 `+`，没有任何待处理内容。运行中的总值存在于 `acc` 中，而不是在等待的栈帧中。理论上，实现了 TCO 的运行时可以在此深度下以常数栈空间执行。

现在重复同样的压力测试：

```js
sumTR(100000); // 仍然可能抛出 RangeError！
```

即使尾递归结构正确，许多 JavaScript 运行时仍然会为每次调用分配新的栈帧，并在深度较大时抛出异常。这让那些期望 TCO 是通用保证的开发者感到惊讶。[ECMAScript 2015](https://262.ecma-international.org/6.0/#sec-tail-position-calls) 正式规定了严格模式下的恰当尾调用，但大多数引擎从未一致地采用该特性。有些引擎曾实现过它，后来由于性能回退又撤销了。其他引擎则从未实现过。结果是，即使代码结构正确支持 TCO，你也不能假设尾递归在生产环境的 JavaScript 中是栈安全的。

## 关于斐波那契的说明

[斐波那契](https://en.wikipedia.org/wiki/Fibonacci_number)是教科书级别的递归示例，它确实也会遇到栈限制，但它还有第二个问题使其更糟糕：**指数级时间复杂度**。

```js
function fib(n) {
  if (n <= 1) return n;
  return fib(n - 1) + fib(n - 2);
}
```

每次调用分支为两个调用，因此总调用次数以 O(2ⁿ) 增长。`fib(30)` 已经产生超过一百万次调用；`fib(50)` 则达到数百亿次。在浏览器中，这会在达到栈限制之前就导致标签页卡死，这使得失败模式看起来与栈溢出完全一样，但根本原因截然不同。

斐波那契的尾递归版本：

```js
function fibTR(n, a = 0, b = 1) {
  if (n === 0) return a;
  if (n === 1) return b;
  return fibTR(n - 1, b, a + b);
}
```

这个版本以线性时间运行，但由于同样的 TCO 不确定性，在大 `n` 时仍然存在栈溢出风险。指数级版本对本讨论来说是干扰项，因为它因完全不同的原因而失败：栈溢出和指数爆炸是两个独立的问题。它们从外部看起来一样（页面卡死或崩溃），但需要完全不同的修复方法。

## 运行时的现实（截至写作时）

截至写作时（2026 年 5 月），恰当尾调用优化的支持在 JavaScript 运行时之间并不可靠。

| 运行时 | 引擎 | 可依赖的恰当尾调用？ | 实际结论 |
|--------|------|---------------------|---------|
| Chrome | V8 | 否 | 不要指望尾递归栈安全。 |
| Node.js | V8 | 否 | 尾递归代码仍可能溢出。 |
| Deno | V8 | 否 | 与 Node/Chrome 相同的操作预期。 |
| Firefox | SpiderMonkey | 否 | 不要将尾递归视为安全保障。 |
| Safari | JavaScriptCore | 不一致——JSC 在多个版本中曾实现后又撤销了 TCO | 不要依赖它；各版本间行为变化足够大，不能作为稳定保证。 |
| Bun | 基于 JavaScriptCore | 依赖于引擎，不是跨运行时保证 | 在确切版本上验证；不要假定通用行为。 |

关键在于可移植性。尾递归是函数结构的属性，而栈复用是运行时实现的属性。即使一个引擎在某个版本中表现更好，生产环境中的 JavaScript 通常覆盖多个目标，正确性不应依赖于特定优化器的行为。一个函数在形式上可以完全是尾递归的，但在用户实际运行的环境中仍然每次调用都消耗栈空间。

## 生产代码的更好模式

每个递归函数都可以用迭代方式重写，当输入深度可能增长时，这通常是生产环境中最安全的选择。迭代不依赖于运行时优化来保证栈安全，因为它每一步不消耗栈帧。这并不意味着放弃递归的思维模型。你仍然可以编写概念上是递归的代码，但使用显式栈或**蹦床**（trampoline）来管理控制流，而不触及物理限制。

```js
function sumIter(n) {
  let acc = 0;
  for (let i = n; i > 0; i--) acc += i;
  return acc;
}

sumIter(1000000); // 没有递归栈增长
```

---

## 蹦床模式

如果你想保留递归结构以提高可读性，但又需要避免栈增长，可以使用蹦床：一个循环，重复调用一个返回最终结果或另一个待调用函数的函数。

```js
function trampoline(fn) {
  let result = fn;
  while (typeof result === 'function') {
    result = result();
  }
  return result;
}

function sumTrampoline(n, acc = 0) {
  if (n === 0) return acc;
  return () => sumTrampoline(n - 1, acc + n);
}

trampoline(() => sumTrampoline(100000)); // 没有栈溢出，精神上仍然是尾递归
```

蹦床以额外的函数分配和调度开销换取了栈安全性，因此在保留递归结构比原始性能更重要时最为有用。

这种方法能够扩展，而不依赖于运行时的尾调用行为——当输入深度可能增长时，这正是你所需要的。如果递归结构能提高特定问题的可读性，这些技术让你能够保留那种思维模型，同时使用显式权衡而非隐式运行时假设。

一个有用的经验法则是：将递归保留用于可控的小且有限的深度，**一旦深度由用户驱动、数据驱动或在操作上不确定，立即切换到迭代控制流**。对于热点路径，对两种风格进行基准测试，但不要将正确性建立在假设的 TCO 之上。

---

## 实用检查清单

- 永远不要在生产环境的关键路径上假设 JavaScript 有 TCO。
- 使用现实的上限进行测试，而不是玩具输入规模。
- 当深度可能增长时，优先选择迭代实现。
- 将递归视为可读性工具，而不是栈安全保证。

---

## 结论

递归本身不是敌人，未经验证的运行时假设才是。尾递归结构并不会自动让 JavaScript 变得栈安全，而正是这个差距导致了生产环境中许多"在我机器上能跑"的意外。

在递归确实能提高清晰度且深度确实有限时使用它。当深度可能增长或输入超出你的控制时，优先选择使栈行为显式且可移植的迭代设计。
