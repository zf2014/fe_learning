## 背景知识

V8 的优化策略是 **"假设 + 内联缓存"**。V8 通过 Hidden Class（Map）描述对象的"形状"，相同形状的对象共享 Map，内联缓存（Inline Cache, IC）可以高效工作。你的代码模式决定了 V8 能否做出正确假设——代码越"规矩"，V8 的优化越激进。

## 一、保持对象形状（Shape/Map）稳定

### 原理

V8 通过 Hidden Class（Map）描述对象的形状。相同形状的对象共享同一个 Map，内联缓存（IC）可以高效工作。形状不稳定会导致 IC 失效、去优化（deoptimization）。

### 规则

- **始终以相同顺序初始化所有属性**
- **不要动态添加/删除属性**（会把对象推入字典模式）
- 需要可选字段？用 `null` 或 `undefined` 占位

### 示例

#### ❌ 同一个构造函数产生的对象形状不一致

```javascript
function BadRect(width, height) {
  this.width = width;
  if (height !== undefined) {
    this.height = height; // 有时有两个属性，有时只有一个
  }
}
new BadRect(10);      // Map: { width }
new BadRect(10, 20);  // Map: { width, height }  → 不同的 Map
```

#### ✅ 始终以相同顺序声明所有属性

```javascript
function GoodRect(width, height = 0) {
  this.width = width;
  this.height = height; // 始终有相同的形状
}
new GoodRect(10);      // Map: { width, height }
new GoodRect(10, 20);  // Map: { width, height }  → 同一个 Map ✓
```

#### ❌ 动态添加属性 → Map 不断 transition，最终退化为字典模式

```javascript
function handleClick(e) {
  const data = {};
  if (condition1) data.x = 1;
  if (condition2) data.y = 2;
  if (condition3) data.z = 3;
}
```

#### ✅ 固定形状

```javascript
function handleClick(e) {
  const data = { x: null, y: null, z: null };
  if (condition1) data.x = 1;
  if (condition2) data.y = 2;
  if (condition3) data.z = 3;
}
```

## 二、保持数组元素类型一致

### 原理

V8 对数组有多种内部表示（Elements Kinds）：`PACKED_SMI_ELEMENTS` → `PACKED_DOUBLE_ELEMENTS` → `PACKED_ELEMENTS` → `HOLEY_*`。每种转换都是 **单向退化**，不可逆转。

```
转换方向（只能向右，不可逆）：

SMI_ELEMENTS → DOUBLE_ELEMENTS → ELEMENTS → HOLEY_SMI → HOLEY_DOUBLE → HOLEY_ELEMENTS
   (最快)                                              (最慢)
```

### 规则

- **不要在同一个数组中混合不同类型**
- **不要创建稀疏数组**（跳过索引赋值）
- 数组操作用 `push`/`pop`，不要 `delete arr[i]`（会制造 hole）

### 示例

#### ❌ 混合类型 → V8 退化到最通用的表示

```javascript
const arr = [1, 2, 3];           // PACKED_SMI_ELEMENTS ✓
arr.push(4.2);                   // → PACKED_DOUBLE_ELEMENTS（退化！）
// 现在即使再 push 整数，也无法回到 SMI_ELEMENTS
```

#### ❌ 创建稀疏数组 → 变成 HOLEY

```javascript
const arr2 = [];
arr2[1000] = 1;                  // → HOLEY_SMI_ELEMENTS（有大量"洞"）
```

#### ❌ 混入非数值类型

```javascript
const arr3 = [1, 2, 3];          // PACKED_SMI_ELEMENTS
arr3.push("hello");              // → PACKED_ELEMENTS（最通用，包含原型链检查）
```

#### ✅ 保持类型一致

```javascript
const smiArr = [1, 2, 3, 4];       // PACKED_SMI_ELEMENTS
const doubleArr = [1.1, 2.2, 3.3]; // PACKED_DOUBLE_ELEMENTS
const objArr = [{}, {}, {}];       // PACKED_ELEMENTS
```

## 三、让数字留在 Smi 范围内

### 原理

Smi（Small Integer）范围内的整数直接编码在 tagged pointer 中，零堆分配。超出范围变成 HeapNumber，每次运算都可能触发堆分配。

- 64 位平台（指针压缩）Smi 范围：**[-2³⁰, 2³⁰ - 1] ≈ ±10.7 亿**
- 32 位平台：**[-2³⁰, 2³⁰ - 1] ≈ ±10.7 亿**
- 64 位平台（无指针压缩）：**[-2³¹, 2³¹ - 1] ≈ ±21.5 亿**

### 值类型对照

```
-Infinity       → HeapNumber
-(2**30)-1      → HeapNumber
-(2**30)        → Smi（指针压缩下的边界）
-0              → HeapNumber（注意！-0 不在 Smi 范围）
 0              → Smi
 4.2            → HeapNumber（小数）
 42             → Smi
 2**30 - 1      → Smi
 2**30          → HeapNumber（溢出）
 Infinity       → HeapNumber
 NaN            → HeapNumber
```

### 真实案例：React 性能断崖

React Fiber 架构中每个 fiber 节点有一个 `expirationTime` 字段。当这个值超出 Smi 范围时，该字段从 Smi 变为 MutableHeapNumber，导致 Map 变更，所有相同形状对象的 IC 全部失效。React 团队最终将时间模型从数字改为位掩码来避免这个问题。

详见：https://v8.dev/blog/react-cliff

### 规则

- 热路径中的计数器、索引确保在 Smi 范围内
- 对于大 ID（如雪花 ID），用字符串而非数字
- 位运算自动截断为 32 位整数，可以帮助保持 Smi

### 示例

```javascript
// ❌ 大数字超出 Smi → HeapNumber → 堆分配
const id = 2147483648;  // 超出 Smi
const sum = id + 1;     // HeapNumber + Smi → 新 HeapNumber（堆分配）

// ✅ 大 ID 用字符串
const id = "2147483648";

// ✅ 热循环中的计数器是 Smi，高效
for (let i = 0; i < arr.length; i++) { ... }
```

## 四、保持函数调用的单态性（Monomorphic）

### 原理

V8 的内联缓存（IC）为每个调用点记录目标的 Map。如果一个调用点总是看到相同形状的对象，它就是单态的（monomorphic），V8 会内联属性访问。如果看到不同形状的对象，IC 退化：

```
单态 (monomorphic)     → 1 个 Map 记录，最快
多态 (polymorphic)     → 2-4 个 Map 记录，中等
超多态 (megamorphic)   → 4+ 个 Map 记录，退化为字典查找
```

### 规则

- 在面向对象代码中，确保同类实例有相同属性
- 避免将结构完全不同的对象传给同一个函数

### 示例

#### ❌ 同一个函数处理不同形状的对象 → 多态退化

```javascript
function getX(obj) {
  return obj.x;
}

getX({ x: 1 });                    // IC 记录 Map1
getX({ x: 1, y: 2 });             // IC 记录 Map2 → polymorphic
getX({ x: 1, y: 2, z: 3 });      // IC 记录 Map3 → megamorphic
// 此时属性访问退化为字典查找
```

#### ✅ 相同形状的对象 → 单态

```javascript
function getX(obj) {
  return obj.x;
}

getX({ x: 1, y: null });          // MapA
getX({ x: 2, y: null });          // MapA（同一个 Map）→ 单态 ✓
```

## 五、不要在热路径上改变对象结构

### 原理

频繁增删属性会把对象推入字典模式（slow mode），属性访问从 O(1) 的索引查找变成哈希表查找。属性类型变更也会触发 Map transition。

### 规则

- 不 `delete` 对象属性，用 `null` 替代
- 对象字段保持类型一致（不要在 Smi 和 Double 之间切换）

### 示例

#### ❌ 对象池模式中频繁 delete

```javascript
function resetPoolObject(obj) {
  for (const key of Object.keys(obj)) {
    delete obj[key];  // 每次 delete 都可能改变 Map → 字典模式
  }
}
```

#### ✅ 赋值为 null 保持形状

```javascript
function resetPoolObject(obj) {
  for (const key of Object.keys(obj)) {
    obj[key] = null;  // 不改变形状 ✓
  }
}
```

#### ❌ 属性赋值改变字段类型 → Map 变更

```javascript
const point = { x: 1, y: 2 };      // Map: { x: Smi, y: Smi }
point.x = 1.5;                      // Map: { x: Double, y: Smi } → 形状变了！
```

#### ✅ 从一开始就用一致类型

```javascript
const point = { x: 1.0, y: 2.0 };  // Map: { x: Double, y: Double }
point.x = 1.5;                      // 同样的 Map ✓
```

## 六、字符串操作意识

### 原理

字符串在 V8 堆上分配。V8 会做字符串驻留（interning），相同内容的字符串字面量可能共享内存。但动态拼接产生 `ConsString`（延迟拼接的树结构），多次访问会触发扁平化。

### 规则

- 大量字符串拼接用数组 `join` 替代 `+=`
- 复用字符串字面量可以受益于驻留优化

### 示例

#### ❌ 循环中反复拼接 → 创建大量中间 ConsString

```javascript
let result = '';
for (let i = 0; i < 1000; i++) {
  result += 'x'; // 每次创建新的 ConsString
}
```

#### ✅ 数组 join 一次性拼接

```javascript
const parts = [];
for (let i = 0; i < 1000; i++) {
  parts.push('x');
}
const result = parts.join(''); // 一次分配
```

## 七、总结清单

| 规则 | 原因 | 性能影响 |
|---|---|---|
| 始终以相同顺序初始化所有属性 | 共享 Map，IC 有效 | ⭐⭐⭐ |
| 数组保持单一类型，不要制造 hole | 避免元素类型退化 | ⭐⭐⭐ |
| 热路径上的数字尽量在 ±10 亿内 | 保持 Smi，零堆分配 | ⭐⭐⭐ |
| 同一函数处理相同形状的对象 | 保持 IC 单态 | ⭐⭐ |
| 不 delete 对象属性，用 null 替代 | 避免字典模式 | ⭐⭐ |
| 对象字段保持类型一致 | 避免 Map transition | ⭐⭐ |
| 大量字符串拼接用数组 join | 减少 ConsString 中间对象 | ⭐⭐ |
| 避免在热路径上用 `arguments` 对象 | 导致去优化 | ⭐ |

> **重要提醒**：不要过度优化。这些规则在热路径（被调用成千上万次的代码）上才有意义。普通的业务代码，可读性优先。先用性能分析工具找到瓶颈，再有针对性地优化。

## 八、如何验证和调试

### 查看字节码

```bash
node --print-bytecode script.js
```

### 查看优化/去优化日志

```bash
node --trace-opt --trace-deopt script.js
```

### 查看对象的 Map 和属性

```bash
node --allow-natives-syntax -e '
  const obj = { x: 1 };
  %DebugPrint(obj);  // 输出 Map 信息、属性布局等
'
```

### 查看数组元素类型

```bash
node --allow-natives-syntax -e '
  const arr = [1, 2, 3];
  %DebugPrint(arr);  // 输出 elements kind: PACKED_SMI_ELEMENTS
'
```

### Chrome DevTools

1. 打开 Performance 面板录制
2. 关注 Bottom-Up 视图中的热点函数
3. 使用 Memory 面板的 Heap Snapshot 检查对象分布
