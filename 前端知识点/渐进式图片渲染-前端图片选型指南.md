# 渐进式图片渲染 — 前端图片选型指南

> 基于 Jake Archibald《The present and potential future of progressive image rendering》(2025.10) 的核心观点，结合前端实际场景给出的选型方案。

---

## 一、通用原则

1. **AVIF > WebP > JPEG** 是当前主流优先级。AVIF 压缩率最高、解码最快，是性价比之王。
2. **图片体积本身是第一优先级**。渐进式渲染只在"慢连接 + 恰好收到部分数据"的窄窗口下才有感知，不能用它替代对体积的优化。
3. **渐进式渲染 ≠ 零成本**。即使是 JPEG，渐进编码也会增加约 40% 解码时间（绝对值小但存在）；AVIF 的 layered 模式有额外的文件体积开销。

---

## 二、各格式渐进式渲染现状

| 格式 | 渐进式渲染表现 |
|---|---|
| **JPEG** | 默认支持渐进编码。约 1.2 kB 即可开始渲染低分辨率，Firefox/Chrome 会做平滑处理，Safari 呈现块状。解码耗时多约 40%，但绝对值很小（~1.3ms），是合理的折中。文件体积开销几乎为零。 |
| **PNG** | 交织模式文件体积膨胀严重，**不推荐**。 |
| **WebP** | Firefox/Chrome 从上到下渲染，但需接收约 28% 数据才出图；Safari **完全不支持**渐进渲染。 |
| **AVIF** | 标准 AVIF **不支持**渐进渲染。有实验性的 `--layered` 模式可创建两层（低分辨率预览 + 全分辨率），仅 Chrome 支持，预览质量差，编码参数也难以调优。 |
| **JPEG XL** | Safari 是目前唯一支持的浏览器，但**不支持**渐进渲染，解码速度是 AVIF 的 2.5 倍。实验性的 `--progressive_dc` + jxl-oxide 解码器可以实现多阶段渐进渲染，但速度不够用于浏览器。 |

## 三、按场景的选型方案

### 场景 1：首屏 Hero 图 / LCP 图片

**目标：** 最快显示、最大视觉冲击

| 决策 | 理由 |
|---|---|
| 主格式：AVIF（`<picture>` fallback WebP → JPEG） | 最小体积 = 最快传输，解码也快 |
| 尺寸控制：`srcset` + `sizes` 提供多分辨率 | 避免小屏下载 2x 大图，比指望渐进式截断靠谱得多 |
| preload：`<link rel="preload" as="image">` | 提前发起请求，比渐进渲染的实际收益大 |
| 不建议依赖渐进式渲染 | 首屏图片通常 < 100 kB（AVIF），下载本身很快，渐进式几乎无感知 |

```html
<picture>
  <source srcset="hero-400.avif 400w, hero-800.avif 800w, hero-1200.avif 1200w"
          type="image/avif">
  <source srcset="hero-400.webp 400w, hero-800.webp 800w, hero-1200.webp 1200w"
          type="image/webp">
  <img srcset="hero-400.jpg 400w, hero-800.jpg 800w, hero-1200.jpg 1200w"
       sizes="100vw" alt="hero" loading="eager" fetchpriority="high">
</picture>
```

---

### 场景 2：内容图 / 博客正文配图

**目标：** 体积与体验平衡，可接受短暂空白

| 决策 | 理由 |
|---|---|
| 主格式：AVIF + JPEG fallback | 内容图数量多，AVIF 省带宽效果最显著 |
| 懒加载：`loading="lazy"` | 视口外图片不抢占带宽 |
| **JPEG 用渐进编码** | 博客正文图通常较大（照片类），渐进 JPEG 在慢速下的体验提升覆盖面广，体积开销几乎为零，解码代价可忽略 |
| AVIF 不折腾 layered 模式 | 实验性太强、只支持 Chrome、质量难调，投入产出不合理 |

```html
<picture>
  <source src="article-photo.avif" type="image/avif">
  <source src="article-photo.webp" type="image/webp">
  <img src="article-photo.jpg" loading="lazy" alt="...">
</picture>
```

编码时对 JPEG 使用渐进模式（Squoosh 默认开启、`jpegtran -progressive`、Sharp 的 `{ progressive: true }`）。

---

### 场景 3：缩略图 / 列表页 / 卡片图

**目标：** 体积极小、加载密集

| 决策 | 理由 |
|---|---|
| AVIF 低质量即够用 | 卡片图通常 < 50px 显示，AVIF 压到几 kB 即可清晰 |
| **不需要渐进式** | 体积太小（通常 < 20 kB），一次性下载完成，渐进式无意义 |
| 可考虑 LQIP（Low Quality Image Placeholder） | 极低质量缩略图做骨架屏占位，比依赖格式原生渐进式更可控 |

---

### 场景 4：图库 / 高清大图（摄影集、作品集）

**目标：** 高分辨率、可能慢加载

| 决策 | 理由 |
|---|---|
| AVIF 为主格式 | 高分辨率照片体积差异最大，AVIF 优势最明显 |
| **用 JS 模拟渐进式**：先加载极低质量小图（1-2 kB AVIF 缩略图 + CSS blur），再加载完整图 | 比任何原生渐进式方案都可控——质量你自己定、开销你自己定、所有浏览器都能用 |
| 不用 AVIF layered / JPEG XL | 前者实验性太强，后者浏览器不支持渐进渲染 |

```css
/* LQIP 模糊占位 */
.img-placeholder {
  filter: blur(20px);
  transition: opacity 0.3s;
}
.img-placeholder.loaded {
  filter: none;
  opacity: 1;
}
```

---

### 场景 5：图标 / Logo / 简单图形

**目标：** 矢量优先，极致清晰

| 决策 | 理由 |
|---|---|
| SVG | 矢量无损、任意缩放、体积小 |
| 不使用位图格式 | 除非有照片质感的图标，否则无需栅格化 |

---

### 场景 6：动画 / 动图

| 决策 | 理由 |
|---|---|
| 短动画：Video（`<video muted autoplay loop playsinline>`） | 体积远小于 GIF |
| AVIF 动画序列 | 如果浏览器支持，比 WebP 动画更小 |
| GIF 仅作 fallback | 体积大、质量差，只做兜底 |

---

## 四、决策流程图

```
需要图片？
├─ 是图标/Logo/简单图形 → SVG
├─ 是动画
│   ├─ 短循环 → <video> (mp4/webm)
│   └─ 兼容需求 → GIF fallback
└─ 是照片/栅格图
    ├─ 显示尺寸 < 100px（缩略图）
    │   └─ AVIF 低质量，无需渐进式
    ├─ 首屏 LCP 图片
    │   └─ AVIF + srcset 多分辨率 + preload，无需渐进式
    ├─ 正文配图（中等大小）
    │   └─ AVIF + JPEG 渐进编码 fallback
    └─ 超大高清图（> 200 kB）
        └─ AVIF + JS LQIP（blur placeholder）策略
```

---

## 五、总结要点

1. **AVIF 是当前最优选**——体积最小、解码最快，渐进式渲染的缺失在实际体验中影响不大。
2. **JPEG 渐进编码是无成本保险**——对 fallback JPEG 开启 `progressive`，几乎零开销换取慢速下的渐进体验。
3. **不要指望用单张大图 + 渐进式渲染替代响应式图片**——`srcset`/`sizes` 才是正解，反馈延迟会让截断下载失效。
4. **需要渐进体验时，自己造比格式原生更靠谱**——LQIP（缩略图 + CSS blur）完全可控、全浏览器兼容、零格式依赖。
